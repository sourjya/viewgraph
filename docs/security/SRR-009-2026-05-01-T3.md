## New Findings

### S3-9: Shell injection via `filePath` in `check-bundle-size.js` (MEDIUM)

- **File**: `scripts/check-bundle-size.js:51`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  const gzipSize = parseInt(execSync(`gzip -c "${filePath}" | wc -c`).toString().trim(), 10);
  ```
- **Attack scenario**: `filePath` is constructed from `path.resolve(ROOT, budget.path)` where `budget.path` comes from `bundlesize.config.json`. If an attacker can modify `bundlesize.config.json` (e.g., via a malicious PR or compromised dependency), they can inject shell commands via a crafted `path` value like `"; rm -rf / #`. The double-quote escaping is insufficient — a value containing `"$(malicious)"` would execute.
- **Exploitability**: Low in practice — requires write access to `bundlesize.config.json` which is committed to the repo. However, this script runs in CI (`ci.yml` step "Bundle size check"), so a malicious PR modifying the config file would execute arbitrary commands in the CI runner.
- **Recommended fix**: Use `execFileSync` with array arguments instead of string interpolation:
  ```js
  import { execFileSync } from 'node:child_process';
  const gzipOutput = execFileSync('sh', ['-c', `gzip -c -- "$1" | wc -c`, '--', filePath], { encoding: 'utf-8' });
  ```
  Or better, use Node's `zlib.gzipSync` to compute gzip size without shelling out:
  ```js
  import { gzipSync } from 'node:zlib';
  const gzipSize = gzipSync(readFileSync(filePath)).length;
  ```
- **Regression test exists**: No.

### S16-3: Vitest plugin `label` parameter used in filename without full sanitization (LOW)

- **File**: `packages/vitest/index.js:143-144`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  const suffix = label ? `-${label.replace(/[^a-z0-9]+/gi, '-').toLowerCase()}` : '';
  const filename = `viewgraph-vitest-${ts}${suffix}.json`;
  ```
- **Attack scenario**: The `label` parameter is sanitized with a regex that replaces non-alphanumeric characters with hyphens. This is adequate for most cases. However, an extremely long label (e.g., 10,000 characters) would produce a very long filename that could exceed filesystem limits (255 bytes on ext4). The `writeFileSync` call would throw, but the error is unhandled and would crash the test.
- **Exploitability**: Very low — the caller controls the label and this runs in a test environment. Not a security vulnerability per se, but a robustness issue.
- **Recommended fix**: Cap the suffix length:
  ```js
  const suffix = label ? `-${label.replace(/[^a-z0-9]+/gi, '-').toLowerCase().slice(0, 50)}` : '';
  ```
- **Regression test exists**: No.

## Focus Area Analysis

### 1. Gateway Proxy Security (server/src/clusters/gateway.js)

**Question**: Does the Proxy leak internal state?

**Finding**: No. The Proxy is well-designed:

- **Trap scope**: Only the `get` trap is used, and only for the `tool` property. All other property accesses pass through to the real server via `target[prop]`. Functions are correctly `.bind(target)` to preserve `this` context.
- **Registry isolation**: The `registry` Map is a local variable in `createToolProxy()`, not exposed on the proxy object. It's only accessible via the returned `{ proxy, registry }` tuple.
- **No prototype pollution**: The Proxy doesn't intercept `set`, `defineProperty`, or `getPrototypeOf` traps. An attacker cannot inject properties into the server via the proxy.
- **Action parameter validation**: The gateway handler validates `params.action` against the registry (`registry.get(params.action)`). Unknown actions return an error, not a crash.
- **Context forwarding**: MCP request context is correctly forwarded to sub-tool handlers (Fix 4).
- **Zod safety**: `safeOptional()` correctly handles `ZodOptional`, `ZodNullable`, and `ZodEffects` types without double-wrapping. The `_def.typeName` check is the correct approach for cross-module-boundary Zod type detection.

**Verdict**: ✅ Clean. No internal state leakage. The Proxy pattern is correctly scoped.

### 2. Vitest Plugin Path Traversal (packages/vitest/index.js)

**Question**: Does `captureDOM` write to arbitrary paths?

**Finding**: Partially mitigated.

- **Default path**: `DEFAULT_DIR` is `process.cwd() + '/.viewgraph/captures/'` — safe, within the project.
- **Custom path**: `options.capturesDir` is accepted without validation. A caller could pass `capturesDir: '/etc/'` and the plugin would write there (if permissions allow).
- **Filename construction**: The filename is constructed from timestamp + sanitized label. No user input flows into the directory path unsanitized.
- **`mkdirSync` with `recursive: true`**: Creates the directory if it doesn't exist. This is safe — it doesn't follow symlinks for the final component.
- **Context**: This is a test utility. The caller is a developer writing tests. The `capturesDir` option is explicitly documented. Restricting it would reduce legitimate flexibility.

**Verdict**: ✅ Acceptable risk. The `capturesDir` parameter is developer-controlled in test code, not agent-controlled. No path traversal from untrusted input. The label sanitization (S16-3) is the only minor gap.

### 3. Component Name Extraction (__REACT_FIBER__)

**Question**: Can `__REACT_FIBER__` data be exploited?

**Finding**: No exploitable vector.

- **Read-only access**: `getReactFiber()` reads `el[key]` where `key` starts with `__reactFiber$`. It never writes to the fiber.
- **Data extraction is bounded**: `getReactComponentName()` extracts only `displayName`, `name`, and `_debugSource` (file + line). It does not serialize the entire fiber tree.
- **`_debugSource` exposure**: In dev builds, `_debugSource.fileName` contains the source file path (e.g., `src/components/ProductCard.tsx:42`). This is included in the capture output. This is intentional — it's the feature's purpose (mapping DOM to source). In production builds, `_debugSource` is stripped by React, so no path leakage occurs.
- **No prototype chain walking**: The fiber traversal uses `current = current.return` (the fiber's parent pointer), not `Object.getPrototypeOf()`. No prototype pollution risk.
- **DOM walk cap**: `walkDOM({ max: 3000 })` caps the number of elements visited, preventing DoS on large pages.
- **`buildSelector()` is called per component**: This is the existing selector builder, already reviewed in prior SRRs.

**Verdict**: ✅ Clean. The component collector reads framework internals safely and extracts only names and source paths.

### 4. Shared Zod Params (server/src/utils/shared-params.js)

**Question**: Any Zod schema weakening?

**Finding**: No weakening.

- **`filenameParam`**: `z.string()` — same constraint as before extraction. No `.optional()` added.
- **`urlFilterParam`**: `z.string().optional()` — correctly optional, same as prior inline definitions.
- **`limitParam`**: `z.number().optional().default(20)` — adds a default of 20. This is a tightening, not a weakening. Previously, tools that forgot to set a default could return unbounded results.
- **`fileAParam` / `fileBParam`**: `z.string()` — required, same as before.
- **No `.max()` on strings**: `filenameParam` has no max length. However, `validateCapturePath()` applies `path.basename()` which inherently caps at filesystem limits. The Zod schema doesn't need a max length because the path validation layer handles it.

**Verdict**: ✅ Clean. The extraction is a pure refactor with no schema weakening.

### 5. CI Workflow (.github/workflows/ci.yml)

- **Pinned actions**: `actions/checkout@v4` and `actions/setup-node@v4` use major version tags, not SHA pins. This is standard practice but means a compromised action could affect CI.
- **No secrets used**: The workflow doesn't reference any secrets or tokens. No credential exposure risk.
- **`npm ci`**: Correct — uses lockfile, no arbitrary resolution.
- **`--max-warnings 10`**: ESLint allows up to 10 warnings. This is a quality gate, not a security concern.
- **No artifact upload**: Build outputs are not uploaded or published from CI. No supply chain risk from CI artifacts.

**Verdict**: ✅ Clean. Consider pinning actions to SHA for defense-in-depth (INFO-5).

## INFO Observations

### INFO-1: Gateway proxy is a clean security boundary

The `createToolProxy` + `registerGateways` pattern correctly separates tool registration from tool dispatch. In clustered mode, individual tools are never registered on the real MCP server — only the 6 gateway tools are. This reduces the attack surface from 41 tool schemas to 6, and the gateway validates the `action` parameter before dispatching.

### INFO-2: `log()` helper eliminates stdout pollution risk

The `log()` function in `constants.js` uses `process.stderr.write()` directly, bypassing `console.error()`. This is more robust than `console.error()` because it cannot be accidentally overridden or redirected. All server logging now goes through this single function.

### INFO-3: Shared Zod params improve consistency

Extracting common parameters to `shared-params.js` ensures all tools that accept `filename`, `urlFilter`, or `limit` use identical schemas and descriptions. This eliminates the risk of one tool having a weaker schema than another (e.g., one accepting `limit` as a string while another requires a number).

### INFO-4: `measure-schema-tokens.js` has shell injection via `echo '${input}'`

- **File**: `scripts/measure-schema-tokens.js:33`
- **Code**: `execSync(\`echo '${input}' | node server/index.js 2>/dev/null\`)`
- The `input` variable is a hardcoded JSON string constructed in the same file. No user input flows into it. Not exploitable, but the pattern is fragile — if the JSON ever contains single quotes, the shell command breaks. Consider using `child_process.spawn` with stdin pipe instead.

### INFO-5: CI actions use major version tags, not SHA pins

`actions/checkout@v4` and `actions/setup-node@v4` could be pinned to specific commit SHAs for defense-in-depth against supply chain attacks on GitHub Actions. This is a best practice for high-security projects but not a vulnerability for a localhost dev tool.
