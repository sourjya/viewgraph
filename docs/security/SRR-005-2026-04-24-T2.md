## Fixes Applied

### S4-4: Timing-safe handshake verification (HIGH → FIXED)
- **Before**: `body.response === expected` (timing attack vulnerable)
- **After**: Uses `verify()` which calls `timingSafeEqual`
- **File**: `server/src/auth/middleware.js`

### S4-7: Challenge map bounded (LOW → FIXED)
- **Before**: Unbounded Map, could grow indefinitely under DoS
- **After**: Max 50 pending challenges, oldest evicted
- **File**: `server/src/auth/middleware.js`

### S5-8: JSON.parse guarded (MEDIUM → FIXED)
- **Before**: `JSON.parse(body)` could throw on malformed input
- **After**: try/catch with empty object fallback
- **File**: `server/src/http-receiver.js`

## Accepted Risk

### S4-2: Session key in handshake response
The server includes the session key in the `/handshake` response so the extension can sign requests. Any localhost process can call `/handshake` and receive the key. This is an accepted risk because:
1. The key rotates on every server restart
2. The handshake is localhost-only (127.0.0.1)
3. Without HMAC, the same localhost processes could already inject captures
4. HMAC still provides replay protection (30s timestamp) and session tracking
5. Native messaging (ADR-016) eliminates this entirely by removing HTTP
