## New Findings

### S3-8: `filePath` parameter allows write to `.viewgraph/.session-key` and `config.json` (MEDIUM)

- **File**: `server/src/tools/get-capture.js:40-44`, `server/src/tools/compare-screenshots.js:44-48`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  const resolved = path.resolve(outputPath);
  const projectRoot = path.resolve(capturesDir, '..'); // .viewgraph/
  if (!resolved.startsWith(projectRoot + path.sep) && resolved !== projectRoot) {
    return errorResponse('Error: filePath must be within the project directory');
  }
  ```
- **Attack scenario**: The `filePath` parameter is validated to be within `.viewgraph/` (the parent of `captures/`). An AI agent (or a malicious prompt injection in captured DOM text) could instruct the tool to write to:
  - `.viewgraph/.session-key` — overwrite the HMAC signing key, enabling session hijack when auth is enforced
  - `.viewgraph/config.json` — overwrite project config (urlPatterns, autoAudit flags)
  - `.viewgraph/baselines/` — overwrite golden baselines with poisoned data
  
  The agent would call `get_capture({ filename: "valid.json", filePath: "/path/to/project/.viewgraph/.session-key" })` and the capture content would overwrite the session key file.
- **Exploitability**: Requires the AI agent to be tricked via prompt injection in captured DOM text. The agent must call the tool with a crafted `filePath`. This is a realistic attack vector given that captures contain untrusted page content and the agent processes them.
- **Recommended fix**: Restrict `filePath` to the `captures/` directory (or a dedicated `exports/` directory), not the entire `.viewgraph/` tree:
  ```js
  const projectRoot = path.resolve(capturesDir); // captures/ not .viewgraph/
  ```
  Or better, create a dedicated `.viewgraph/exports/` directory and restrict writes there.
- **Regression test exists**: No test for filePath pointing to `.session-key` or `config.json`.

### S5-11: Native messaging `annotations:resolved` still uses `includes()` URL matching (MEDIUM)

- **File**: `server/src/native-message-handler.js:79`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  if (url && !entry.url?.includes(url)) continue;
  ```
- **Attack scenario**: The HTTP endpoint `/annotations/resolved` was fixed in MRR-008 to use `new URL()` hostname+port matching (resolving S5-10). However, the native messaging handler for the same operation still uses the vulnerable `includes()` pattern. A `url` value of `evil.com` would match any capture whose URL contains `evil.com` as a substring (e.g., `http://localhost:3000?redirect=evil.com`).
- **Exploitability**: Low in practice — native messaging is only accessible from the extension itself, not from web pages. But it's an inconsistency that could become exploitable if native messaging is exposed to other contexts.
- **Recommended fix**: Apply the same URL parsing logic as the HTTP endpoint:
  ```js
  if (url) {
    try {
      const pu = new URL(url.includes('://') ? url : `http://${url}`);
      const eu = new URL(entry.url);
      if (eu.hostname !== pu.hostname || (pu.port && eu.port !== pu.port)) continue;
    } catch { continue; }
  }
  ```
- **Regression test exists**: No.

### S16-2: `source-linker.js` still does not enforce `_MAX_FILE_SIZE` (LOW)

- **File**: `server/src/analysis/source-linker.js:31`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  const _MAX_FILE_SIZE = 200 * 1024;
  // ... later in findSource():
  try { content = await readFile(file, 'utf-8'); } catch { continue; }
  ```
- **Attack scenario**: The `_MAX_FILE_SIZE` constant is defined but never enforced. The `readFile` call reads files of any size. A project with large generated files (e.g., 50MB bundle outputs not in `dist/`) could cause the source linker to read hundreds of MB into memory. This is a DoS vector if `find_source` is called repeatedly.
- **Exploitability**: Low — requires a project with large source files outside the SKIP_DIRS list. The MAX_FILES=5000 cap and depth=10 limit provide some protection.
- **Recommended fix**: Add `stat` check before `readFile`:
  ```js
  const { stat } = await import('fs/promises');
  const s = await stat(file);
  if (s.size > _MAX_FILE_SIZE) continue;
  ```
- **Regression test exists**: No.

## Carried Findings

### S4-3: HMAC body not included in server-side signature verification (HIGH - carried from SRR-005)

- **File**: `server/src/http-receiver.js:163` (auth middleware call with `body: ''`)
- **Status**: Still open. The auth middleware's `validateRequest` receives `body: ''` because auth validation runs before the request body is read. The extension's `auth-sw.js` correctly includes the body hash in signatures. This is a design mismatch.
- **Impact**: When auth is enforced (post-beta), an attacker who can observe a valid signed request could replay it with a different body. The signature would still validate because the server ignores the body hash.
- **Note**: Auth is currently disabled (`requireAuth: false`), so this has zero exploitability today.

## INFO Observations

### INFO-1: MRR-008 significantly improved server code quality

The MRR-008 sprint addressed 22 maintainability findings across 3 phases. Security-relevant improvements include: O(N²) → O(N+M) in consistency-checker, parallel file reads in `readAndParseMulti`, `withCapture()` wrapper pattern established, and `jsonResponse()` migration. These reduce the surface area for future bugs.

### INFO-2: `console.log` eliminated from server source

All 15 `console.log` calls identified in MRR-008 (13.22) have been replaced with `console.error` or removed. This eliminates the MCP stdout pollution risk entirely.

### INFO-3: Container merge preserves security-relevant attributes

The D2Snap-aligned container merge in `traverser.js:230` correctly preserves nodes with `data-*` or `aria-*` attributes, even if they are otherwise merge-eligible wrappers. This prevents loss of `data-testid`, `aria-label`, and other attributes that agents use for element identification.

### INFO-4: TOON compact format properly escapes quotes

The `buildToonManifest()` function in `serializer.js:477` escapes backslashes and double quotes in `axName` values before embedding them in the TOON format. This prevents format injection where a malicious accessible name could break the TOON parser.
