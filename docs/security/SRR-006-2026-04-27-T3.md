## New Findings

### S3-4: Archive fallback reads file path from index.json without validation (MEDIUM)

- **File**: `server/src/tools/get-capture.js:56`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  const archivePath = path.join(archiveDir, entry.filename);
  const content = await readFile(archivePath, 'utf-8');
  ```
- **Attack scenario**: The archive `index.json` stores `entry.filename` as a relative path like `2026-04/viewgraph-host-ts.json`. If the index.json file is tampered with (by local malware or a crafted capture that manipulates the archive process), `entry.filename` could contain `../../etc/passwd` or similar traversal. The `path.join(archiveDir, entry.filename)` does not validate the result stays within `archiveDir`.
- **Exploitability**: Requires write access to `.viewgraph/archive/index.json` (local filesystem). Not remotely exploitable. However, the HTTP receiver's `/captures` endpoint writes captures that trigger `runArchive()`, and the archive module builds index entries from capture metadata - so a malicious capture with a crafted timestamp could influence the month folder name.
- **Recommended fix**: Apply `validateCapturePath()` or equivalent basename + startsWith guard before reading:
  ```js
  const archivePath = path.resolve(archiveDir, entry.filename);
  if (!archivePath.startsWith(path.resolve(archiveDir) + path.sep)) {
    // skip - path traversal in index
    continue;
  }
  ```
- **Regression test exists**: No - the archive fallback path in get-capture is not tested for traversal.

### S3-5: Native message handler uses string concatenation for file paths (MEDIUM)

- **File**: `server/src/native-message-handler.js:82`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  const raw = readFileSync(deps.capturesDir + '/' + entry.filename, 'utf-8');
  ```
- **Attack scenario**: `entry.filename` comes from the indexer, which parses filenames from disk. The indexer uses `path.basename()` during indexing, so this is unlikely to contain traversal sequences in practice. However, the pattern is inconsistent with the rest of the codebase which uses `validateCapturePath()` for all file reads. If the indexer is ever changed to accept filenames from other sources (e.g., native messaging input), this becomes exploitable.
- **Exploitability**: Low - requires indexer to be fed a malicious filename. Currently the indexer only reads from the filesystem.
- **Recommended fix**: Use `validateCapturePath(entry.filename, deps.capturesDir)` for consistency:
  ```js
  const filePath = validateCapturePath(entry.filename, deps.capturesDir);
  const raw = readFileSync(filePath, 'utf-8');
  ```
- **Regression test exists**: No specific path traversal test for this code path.

### S5-9: pushSnapshot and pushScreenshot use hardcoded SERVER_URL (LOW)

- **File**: `extension/entrypoints/background.js:140,162`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  await fetch(`${SERVER_URL}/snapshots`, { ... });  // line 140
  await fetch(`${SERVER_URL}/screenshots`, { ... }); // line 162
  ```
- **Attack scenario**: In multi-project setups, `pushToServer()` correctly discovers the right server via `discoverServer()`, but `pushSnapshot()` and `pushScreenshot()` always send to the default port (9876). If project A runs on 9876 and project B on 9877, screenshots from project B go to project A's server. This is a data routing bug, not a security vulnerability per se, but it means capture data could end up in the wrong project's `.viewgraph/` directory.
- **Recommended fix**: Pass the discovered `serverUrl` to `pushSnapshot()` and `pushScreenshot()` instead of using the hardcoded constant. The `pushToServer()` function already discovers the correct URL - thread it through.

### S4-3: HMAC body not included in signature verification (HIGH - carried from SRR-005)

- **File**: `server/src/http-receiver.js:163`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  const authResult = auth.validateRequest({ method, url, headers: req.headers, body: '' });
  ```
- **Status**: Still open. The body is hardcoded to `''` because auth validation runs before the body is read. This means HMAC signatures don't cover POST bodies - an attacker who can observe a valid signed request can replay it with a different body.
- **Impact**: Reduced because `requireAuth` defaults to `false` (beta mode). When auth is enforced, this becomes exploitable.
- **Recommended fix**: Defer auth validation to after body read, or restructure to read body first then validate. This is a design issue that requires careful refactoring.
