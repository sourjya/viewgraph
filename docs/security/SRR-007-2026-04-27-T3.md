The M19 migration is a **net security improvement**. Moving all HTTP/WS communication from the content script to the service worker eliminates the largest attack surface (content script making direct network calls in the context of potentially hostile pages). The transport-handler whitelist is well-designed. Auth session persistence in `chrome.storage.session` is the correct MV3 pattern.

Two new findings require attention: a variable shadowing bug in native-message-handler.js that breaks the `annotations:resolved` handler entirely, and an overly permissive URL matching pattern in the HTTP receiver's `/annotations/resolved` endpoint.

## New Findings

### S3-6: Variable shadowing breaks native messaging `annotations:resolved` handler (HIGH)

- **File**: `server/src/native-message-handler.js:77-89`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  // Line 77: outer scope
  const resolved = [];
  for (const entry of entries) {
    // ...
    try {
      const filePath = path.join(deps.capturesDir, path.basename(entry.filename));
      const resolved = path.resolve(filePath);  // Line 84: SHADOWS the array
      if (!resolved.startsWith(path.resolve(deps.capturesDir) + path.sep)) continue;
      const raw = readFileSync(resolved, 'utf-8');
      const capture = JSON.parse(raw);
      for (const ann of (capture.annotations || [])) {
        if (ann.resolved) resolved.push({ filename: entry.filename, ...ann });
        // Line 89: resolved.push() calls String.prototype.push - TypeError
      }
    } catch { /* skip unreadable - SILENTLY SWALLOWS THE TypeError */ }
  }
  return { annotations: resolved }; // Always returns empty array
  ```
- **Attack scenario**: This is not a security vulnerability per se, but a functional bug that silently breaks the native messaging resolved annotations feature. The `catch {}` swallows the `TypeError: resolved.push is not a function` error, so the handler always returns `{ annotations: [] }`. Users on native messaging transport never see resolved annotations.
- **Impact**: Functional breakage of native messaging annotation resolution. The HTTP endpoint (`GET /annotations/resolved`) is unaffected and works correctly.
- **Severity**: HIGH (functional breakage of a security-relevant feature - resolution tracking)
- **Recommended fix**: Rename the inner variable:
  ```js
  const resolvedPath = path.resolve(filePath);
  if (!resolvedPath.startsWith(path.resolve(deps.capturesDir) + path.sep)) continue;
  const raw = readFileSync(resolvedPath, 'utf-8');
  ```
- **Regression test exists**: No - the existing test likely doesn't exercise the annotations:resolved case with actual capture files containing resolved annotations.

### S5-10: Overly permissive URL matching in `/annotations/resolved` endpoint (MEDIUM)

- **File**: `server/src/http-receiver.js:429`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  if (!capture.metadata?.url?.includes(pageUrl) && !pageUrl.includes(capture.metadata?.url)) continue;
  ```
- **Attack scenario**: The bidirectional `includes()` check is overly permissive. If `pageUrl` is `localhost:3000` and a capture's URL is `localhost:3000/admin/secret-dashboard`, the check passes because `capture.metadata.url.includes(pageUrl)` is true. More critically, if `pageUrl` is `evil.com` and a capture URL is `http://localhost:3000?redirect=evil.com`, the check passes because `capture.metadata.url.includes(pageUrl)` is true. This could leak annotation data across pages.
- **Concrete scenario**: A malicious page at `evil.com` could craft a request to `GET /annotations/resolved?url=evil.com`. If any capture's URL contains `evil.com` as a substring (e.g., a redirect parameter), that capture's resolved annotations would be returned. Since the server binds to localhost, this requires the malicious page to make a `fetch('http://127.0.0.1:9876/annotations/resolved?url=evil.com')` request - which is possible due to CORS `*`.
- **Recommended fix**: Use hostname+port matching instead of substring includes:
  ```js
  function urlMatchesPage(captureUrl, pageUrl) {
    try {
      const a = new URL(captureUrl);
      const b = new URL(pageUrl);
      return a.hostname === b.hostname && a.port === b.port && a.pathname === b.pathname;
    } catch { return false; }
  }
  ```
- **Regression test exists**: No specific test for URL matching edge cases on this endpoint.

### S1-4: Content script still bundles transport.js with direct fetch() capability (MEDIUM)

- **File**: `extension/lib/discovery.js:16`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  import * as transport from '#lib/transport.js';
  ```
- **Attack scenario**: `discovery.js` imports `transport.js` to call `transport.init()` and `transport.reset()`. Since `discovery.js` is imported by `constants.js` (which is imported by the content script), the entire `transport.js` module - including its `fetch()` calls, WebSocket creation, and auth module - is bundled into the content script. This undermines NFR-4 ("content script has zero direct HTTP/WebSocket connections to the server") because the code is present even if not called at runtime.
- **Impact**: The content script bundle contains `fetch()` and `WebSocket` code that could theoretically be invoked if a code path is missed during migration. The actual runtime behavior is correct (sidebar uses `transport-client.js`), but the dead code in the bundle is a maintenance hazard and increases attack surface.
- **Recommended fix**: Have `discovery.js` send `init`/`reset` commands to the SW instead of importing transport.js directly:
  ```js
  // Instead of transport.init(url), send a message:
  chrome.runtime.sendMessage({ type: 'vg-transport-init', url: response.url });
  ```
  Or create a minimal `transport-lifecycle.js` that only exports `init()` and `reset()` without the fetch/WS code.
- **Regression test exists**: No test verifies that the content script bundle excludes transport.js.

### Q1-3: WS manager event buffer overwrites instead of appending (LOW)

- **File**: `extension/lib/sw/ws-manager.js:149`
- **Confidence**: CONFIRMED
- **Code**:
  ```js
  function _handleEvent(e) {
    try {
      const event = JSON.parse(e.data);
      chrome.storage.local.set({ [WS_EVENTS_KEY]: [event] });
    } catch { /* malformed message */ }
  }
  ```
- **Attack scenario**: Each incoming WS event overwrites the previous one in storage. If two events arrive in rapid succession (e.g., two annotations resolved simultaneously), the first event is lost before the content script's `onChanged` listener fires. The `MAX_EVENTS = 50` constant is defined but never used - there is no circular buffer implementation despite the comment claiming one.
- **Impact**: Potential loss of real-time events during burst scenarios. Low severity because the alarm-based sync (every 30s) will catch up on missed events.
- **Recommended fix**: Either implement the circular buffer as documented, or acknowledge the single-event design and remove the misleading `MAX_EVENTS` constant and "circular buffer" comment.
- **Regression test exists**: No test for rapid event delivery.

### INFO-1: Transport handler whitelist is well-designed

- **File**: `extension/lib/sw/transport-handler.js:20-33`
- **Confidence**: CONFIRMED
- The 14-operation whitelist in `OPS` is a positive security pattern. Unknown operations are rejected with an error. Each operation extracts only the specific args it needs rather than passing the full message through. This prevents content scripts from invoking arbitrary transport methods.

### INFO-2: Auth session in chrome.storage.session is correct MV3 pattern

- **File**: `extension/lib/sw/auth-sw.js:56-68`
- **Confidence**: CONFIRMED
- `chrome.storage.session` is extension-only, not accessible to page scripts, and cleared on browser restart. This is more secure than the previous approach of storing auth state in module-level variables in the content script (which runs in the page's process and is theoretically accessible via prototype pollution).

### INFO-3: Discovery-sw.js port scan uses AbortSignal.timeout correctly

- **File**: `extension/lib/sw/discovery-sw.js:63`
- **Confidence**: CONFIRMED
- Port scan probes use `AbortSignal.timeout(1000)` which prevents hanging connections. The `catch(() => {})` correctly swallows timeout errors for non-responding ports.

### INFO-4: Sync alarms correctly handle server offline state

- **File**: `extension/lib/sw/sync-alarms.js:62-64`
- **Confidence**: CONFIRMED
- When the server is offline, `_pollRequests()` catches the error and clears the badge. This prevents stale badge counts from persisting when the server stops.

### INFO-5: prompts.js reads only from package-relative directory

- **File**: `server/src/prompts.js:18`
- **Confidence**: CONFIRMED
- `PROMPTS_DIR` is computed relative to `import.meta.url` (the package's own directory), not from user input. `readdirSync` only lists `.md` files. `readFileSync` uses `join(PROMPTS_DIR, file)` where `file` comes from `readdirSync` output (filesystem-validated). No path traversal vector exists.

## M19 Security Architecture Assessment

### Improvements

| Before (content script) | After (service worker) | Impact |
|---|---|---|
| Content script makes direct HTTP requests | All HTTP via SW message passing | Eliminates network access from hostile page context |
| Per-tab WebSocket connections | Single WS in SW, ref-counted | Reduces connection count, single auth session |
| Auth state in module variables (content script memory) | Auth in `chrome.storage.session` | Survives SW termination, not in page process memory |
| Duplicate discovery (background + content) | Single discovery in SW | Eliminates duplicate port scanning |
| Polling via `setInterval` (stops when sidebar closes) | `chrome.alarms` (fires even when sidebar closed) | Background sync for resolved annotations |

### Remaining Gaps

1. **Content script still bundles transport.js** (S1-4) - the code is present even if not called
2. **HMAC body verification mismatch** (S4-3) - SW signs body, server ignores it
3. **No auth enforcement** (S4-5) - `requireAuth: false` means all the auth infrastructure is unused
