## Findings

### HIGH

#### S3-2: Auto-learn overwrites existing config keys on fresh write
- **Confidence:** CONFIRMED
- **File:** `server/src/http-receiver.js:282-284`
- **Regression test:** None
- **Evidence:**
  ```js
  const autoConfig = { urlPatterns: [pattern], autoAudit: false, smartSuggestions: true };
  writeFileSync(configFile, JSON.stringify(autoConfig, null, 2));
  ```
  When `shouldLearn` is true and the config file exists but has empty `urlPatterns`, the entire file is replaced with `autoConfig`. Any existing keys the user set via `PUT /config` (e.g., `captureQuality`, `includeScreenshot`, `baselineAutoCompare`) are silently dropped.
- **Attack scenario:** Not a direct attack vector, but a data loss bug triggered by normal usage. User configures settings via sidebar, then captures a page on a fresh project - their settings vanish.
- **TOCTOU note:** There is a read-then-write race between `readFileSync` (line 273) and `writeFileSync` (line 284). A concurrent `PUT /config` between the two calls would be overwritten. In practice this is low risk (single-user localhost tool, sub-millisecond window), but the fix below eliminates it.
- **Remediation:** Merge instead of replace. Read existing config, spread auto-learned keys on top:
  ```js
  let existing = {};
  try { existing = JSON.parse(readFileSync(configFile, 'utf8')); } catch { /* new file */ }
  const merged = { ...existing, urlPatterns: [pattern], autoAudit: existing.autoAudit ?? false, smartSuggestions: existing.smartSuggestions ?? true };
  writeFileSync(configFile, JSON.stringify(merged, null, 2));
  ```
  Effort: 10 min. Risk: None.

### MEDIUM

#### S7-3: innerHTML with hardcoded SVG in settings.js
- **Confidence:** CONFIRMED
- **File:** `extension/lib/sidebar/settings.js:84,91`
- **Regression test:** None
- **Evidence:**
  ```js
  helpLink.innerHTML = '<svg width="12" ...>...</svg>URL mapping docs';
  advLink.innerHTML = '<svg width="12" ...>...</svg>All servers →';
  ```
  Both strings are hardcoded (not server-derived), so this is not directly exploitable. However, it violates the project's own security standard established in SRR-001 (S7-2: "Replace innerHTML with textContent + createElement"). The `sidebar/icons.js` module exists specifically for SVG icon factories.
- **Remediation:** Replace with `createElement('svg')` + `textContent` or import from `icons.js`. Effort: 10 min.

#### S5-4: URL pattern matching uses `String.includes()` - overly permissive
- **Confidence:** CONFIRMED
- **Files:**
  - `extension/lib/sidebar/settings.js:225` - `patterns.some((p) => pageUrl.includes(p))`
  - `extension/lib/constants.js:100` - `pageUrl.includes(pattern)` in `classifyTrust`
  - `extension/lib/discovery.js:148` - `normalizedUrl.includes(pattern)` in `discoverServer`
- **Regression test:** None for this specific bypass
- **Attack scenario:** If a user configures `trustedPatterns: ["myapp.com"]`, then `https://evil.com/redirect?to=myapp.com` would match because `includes()` checks the full URL string including query parameters and path. A malicious page at `https://attacker.com/myapp.com/` would also match.
- **Impact:** The trust gate (F17) could be bypassed for a crafted URL, allowing send-to-agent from an untrusted page. The practical risk is LOW because:
  1. The attacker must know the exact trusted pattern string
  2. Auto-learned patterns are `localhost:PORT` which are hard to embed in a remote URL
  3. The capture still goes through F19 sanitization
- **Remediation:** Parse both the pattern and the URL, then compare hostname+port instead of substring:
  ```js
  function matchesPattern(pageUrl, pattern) {
    try {
      const u = new URL(pageUrl);
      const host = u.hostname + (u.port ? ':' + u.port : '');
      return host === pattern || u.hostname === pattern;
    } catch { return false; }
  }
  ```
  Effort: 30 min (3 call sites). Risk: Low - may break intentionally broad patterns, but those are a security risk anyway.

#### Q3-3: F19 wrapping still missing on 4 tools
- **Confidence:** CONFIRMED
- **Files:**
  - `server/src/tools/compare-captures.js:35-36` - `text: n.text` unwrapped
  - `server/src/tools/compare-baseline.js:73-74` - `text: n.text` unwrapped
  - `server/src/tools/get-latest.js` - returns full capture JSON unwrapped
  - `server/src/tools/get-capture.js` - returns full capture JSON unwrapped
- **Regression test:** None
- **Note:** `get-latest` and `get-capture` return the entire capture JSON, which is harder to wrap field-by-field. The `compare-*` tools are straightforward fixes.
- **Remediation:** Add `wrapCapturedText(n.text)` to compare-captures and compare-baseline. For get-latest/get-capture, add a `_notice` field documenting that text is untrusted. Effort: 15 min.

### LOW

#### S7-4: DOM re-append guards bypass framework cleanup
- **Confidence:** NEEDS VALIDATION
- **File:** `extension/lib/annotation-sidebar.js:384-385,413-414`
- **Evidence:**
  ```js
  // Re-append if removed from DOM (some frameworks clean unknown children)
  if (!badgeEl.parentElement) document.documentElement.appendChild(badgeEl);
  // Re-append host if removed from DOM
  if (hostEl && !hostEl.parentElement) document.documentElement.appendChild(hostEl);
  ```
  These guards re-insert ViewGraph elements into `document.documentElement` if a framework (React, Vue) removes them during reconciliation. This is defensive and necessary for the extension to survive framework DOM cleanup.
- **Risk:** Minimal. The elements are ViewGraph's own shadow host and badge strip - not user content. The `ATTR` data attribute marks them as ViewGraph-owned. A malicious page cannot trigger re-append of arbitrary elements because `badgeEl` and `hostEl` are module-scoped variables set only by `create()`.
- **Recommendation:** No change needed. The pattern is safe. Add a comment noting the security rationale.

#### S5-5: suggestions-ui module state not cleared on destroy
- **Confidence:** CONFIRMED
- **File:** `extension/lib/sidebar/suggestions-ui.js:28` - `let _expandedState = null;`
- **Evidence:** `_expandedState` holds references to `{ wrapper, suggestions, callbacks }`. When the sidebar is destroyed via `annotation-sidebar.js:destroy()`, this module state is not explicitly cleared. The `collapseSuggestions()` export is called from the Escape handler but not from `destroy()`.
- **Impact:** Memory leak only. The stale `_expandedState` holds DOM references to a detached shadow tree. No security impact - the callbacks reference sidebar-internal functions, not page content.
- **Remediation:** Export a `resetSuggestions()` function and call it from `destroy()`:
  ```js
  export function resetSuggestions() { _expandedState = null; }
  ```
  Effort: 5 min.

## Parser Audit (viewgraph-v2.js)

The `parseMetadata` function now reads annotations for count/resolved:

```js
annotationCount: Array.isArray(raw.annotations) ? raw.annotations.length : 0,
resolvedCount: Array.isArray(raw.annotations) ? raw.annotations.filter((a) => a.resolved).length : 0,
```

**Assessment:** Safe.
- `Array.isArray()` guard prevents type confusion (object, string, number all return 0)
- `.filter((a) => a.resolved)` is safe - if `a` is not an object, `a.resolved` is `undefined` which is falsy
- No prototype pollution risk - `raw` comes from `JSON.parse()` which does not set `__proto__`
- No DoS risk - annotations array is bounded by the 5MB payload limit (~50K annotations max, filter is O(n))
