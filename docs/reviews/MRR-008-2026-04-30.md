# MRR-008 - Maintainability Review Report (Full Server Codebase)

**Date:** 2026-04-30
**Baseline:** MRR-007 (2026-04-29)
**Scope:** Full server codebase - `server/src/` (all modules: core, auth, analysis, parsers, tools, utils)
**Reviewer:** Fresh-eyes pass — no reference to prior review history during analysis
**Cross-reference:** Carried findings from MRR-004/005 remain open unless noted

---

## Executive Summary

This is a full-scope server-side review covering ~6,800 LOC across 60 source files. Unlike prior MRRs which tracked incremental regressions, this was a clean-slate read of every file. The codebase has good bones — auth crypto is solid, path traversal protection is consistent, and the analysis modules are well-structured pure functions. However, three systemic problems dominate:

1. **Performance regressions at scale.** `readAndParseMulti` is sequential when it should be parallel (14 callers affected). `consistency-checker.js` has an O(N²×M²) loop. `source-linker.js` has an O(files×searches×lines) algorithm with an unenforced size limit. `indexer.js` sorts the full index on every eviction. These are invisible in dev with a handful of captures and severe in prod with hundreds.

2. **Silent failure cascade.** Six or more catch blocks across the server swallow errors without logging, native-message-handler.js has two stubs that return fake success responses, and `readAndParseMulti` drops failed files with no warning passed to the caller. The AI agent calling tools has no way to know it got a truncated dataset.

3. **Abstraction debt still growing.** The `withCapture()` wrapper needed to unify 25 duplicated error-check patterns was a Phase 1 recommendation from the 2026-04-18 review and still doesn't exist. Annotation extraction is duplicated across 3 tools. `getAnnotationSelector()` doesn't exist yet identical inline fallback chains appear in 4 files. Response format and field naming are still inconsistent across tools.

**New findings:** 22 (5 HIGH, 11 MEDIUM, 6 LOW)
**Carried open findings from MRR-004/005/007:** See REVIEW_LOG
**Resolved since MRR-007:** None (this review does not supersede open items)

## New Findings

### 13.1 — `readAndParseMulti` is sequential; 14 tool callers pay the cost

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Systemic |
| **Files** | `server/src/utils/tool-helpers.js:159-165`, callers: `analyze-journey.js`, `analyze-patterns.js`, `compare-captures.js`, `detect-recurring-issues.js`, `diff-annotations.js`, `generate-spec.js`, `get-fidelity-report.js`, `get-session.js`, `get-session-status.js`, `list-sessions.js`, `visualize-flow.js` (and 3 more) |

**Evidence:** `readAndParse` is awaited inside a `for...of` loop, meaning each file is read and parsed before the next begins. `readAndParsePair` at line 142 already solves this correctly with `Promise.all`. These are sibling functions in the same file.

```js
// current — sequential
for (const filename of filenames) {
  const { ok, parsed } = await readAndParse(filename, capturesDir, level);
  if (ok) results.push({ filename, parsed });
}

// fix — parallel
const settled = await Promise.allSettled(
  filenames.map(f => readAndParse(f, capturesDir, level))
);
settled.forEach((r, i) => {
  if (r.status === 'fulfilled' && r.value.ok)
    results.push({ filename: filenames[i], parsed: r.value.parsed });
});
```

For a tool like `analyze-journey.js` that accepts 20 files, this is a 20× latency penalty for no reason.

**Recommended fix:** One-line change to the function body. Also update return type to include a `warnings` field for files that failed (see 13.4).

**Effort:** 20 minutes
**Regression risk:** None — parallel reads of independent files are identical in output

### 13.2 — O(N²×M²) nested loop in `consistency-checker.js`

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Local |
| **Files** | `server/src/analysis/consistency-checker.js:163-184` |

**Evidence:** The outer loop at line 163 iterates every pair of captures `(i, j)`. Inside, `matchElements()` iterates every element of capture A against every element of capture B. With 100 captures × 1000 elements, worst case is 100×99/2 × 1000×1000 = ~5 billion comparisons.

```js
for (let i = 0; i < captures.length; i++) {
  for (let j = i + 1; j < captures.length; j++) {
    // for each pair, matchElements does O(M²) work
    const matches = matchElements(elemsA, elemsB);
    for (const match of matches) { ... }
  }
}
```

**Recommended fix:** Build a Map indexed by testid/selector once per capture before the outer loop. Cross-capture matching becomes O(N×M) lookups instead of O(N²×M²).

```js
const captureIndex = captures.map(c => {
  const byTestid = new Map(c.elements.filter(e => e.testid).map(e => [e.testid, e]));
  const bySelector = new Map(c.elements.map(e => [e.selector, e]));
  return { ...c, byTestid, bySelector };
});
// then O(N) pairwise with O(1) element lookups
```

**Effort:** 2-3 hours
**Regression risk:** Medium — rewrite of matching logic, needs test coverage

### 13.3 — O(files×searches×lines) in `source-linker.js`; `_MAX_FILE_SIZE` never enforced

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Local |
| **Files** | `server/src/analysis/source-linker.js:31,94-115` |

**Evidence:** Line 31 defines `const _MAX_FILE_SIZE = 200 * 1024` with a `TODO: enforce in readFile loop` comment that was never resolved. The file scan at line 94 reads every file in the source tree (up to 5000 files) sequentially and does full-text line matching per search term. For a project with 5000 files × 5 search terms × 1000 lines, that's 25 million line comparisons, all sequential.

**Recommended fix:**
1. Enforce `_MAX_FILE_SIZE` in the read loop — skip files over 200KB before reading them.
2. Parallelize the per-file reads with `Promise.all` (same fix pattern as 13.1).
3. The early-exit at line 114 (`if high-confidence results >= 5, break`) is correct but runs too late. Add a per-file size pre-check using `fs.stat` before `readFile`.

**Effort:** 1-2 hours
**Regression risk:** Low — adds guards, doesn't change algorithm logic

### 13.4 — Six server-side silent catch blocks swallow errors without logging

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Systemic |
| **Files** | `server/src/analysis/post-capture-audit.js:33,85`, `server/src/auth/middleware.js:61`, `server/src/analysis/source-linker.js:96`, `server/src/utils/tool-helpers.js:186`, `server/src/analysis/spec-generator.js:51` |

**Evidence:** Each location catches an exception and either returns `null`/`{}`/`false` or `continue`s silently. When these fire in production, there is no trace of what failed, what input caused it, or what the error was. Contrast with the WS server at line 81 which correctly logs on disconnect.

Specific cases:
- `post-capture-audit.js:33` — catches parse failure, returns null, no log of which file failed
- `post-capture-audit.js:85` — catches `diffCaptures()` throw, returns `{}`, agent gets no regression data
- `middleware.js:61` — returns null if body/secret missing, no auth rejection logged
- `source-linker.js:96` — silently skips unreadable files; permission errors invisible
- `tool-helpers.js:186` — catch in `buildNotFoundMessage` swallows the error entirely

**Recommended fix:** Every catch block that currently swallows should write one line to `process.stderr`:

```js
} catch (err) {
  process.stderr.write(`[viewgraph] post-capture-audit parse failed: ${err.message}\n`);
  return null;
}
```

This is not a logger abstraction (tracked separately as 13.22) — just literal `process.stderr.write` in each location.

**Effort:** 30 minutes
**Regression risk:** None

### 13.6 — `indexer.js` sorts full index on every eviction

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Files** | `server/src/indexer.js:56-64` |

**Evidence:** When the index exceeds `maxCaptures`, the eviction function sorts all entries by timestamp (O(n log n)) then deletes the oldest. This runs on every new file arrival once the index is at capacity. A `Map` preserves insertion order, so the oldest entry is always first — no sort needed.

```js
// current
const sorted = [...index.entries()].sort((a, b) => a[1].ts - b[1].ts);
while (index.size > maxCaptures) index.delete(sorted.shift()[0]);

// fix — Map preserves insertion order, first key is oldest
while (index.size > maxCaptures) index.delete(index.keys().next().value);
```

This assumes entries are always inserted in chronological order, which is true given the watcher's file-add events.

**Effort:** 5 minutes
**Regression risk:** Low — validate with indexer tests

### 13.16 — `auth/middleware.js` challenge cleanup is O(n) inline on every handshake

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Files** | `server/src/auth/middleware.js:42-48` |

**Evidence:** On every `/auth/challenge` request, the code iterates all pending challenges to delete expired ones. With the max 50-challenge cap this is bounded, but the cap itself means a burst of 50 legitimate rapid handshakes (e.g. CI pipeline running parallel agent sessions) could cause the 51st to fail. **Effort:** 20 minutes
**Regression risk:** Low — auth behavior unchanged

### 13.17 — `session-store.js` — expired sessions accumulate indefinitely

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Files** | `server/src/auth/session-store.js` |

**Evidence:** Sessions are only removed when `validate()` is called and finds them expired. A session created but never validated (e.g. handshake started, never completed) stays in the `Map` forever. On a long-running server with repeated connection attempts, this leaks memory unboundedly.

**Recommended fix:** Add a `setInterval` every 5 minutes to sweep the session map:

```js
setInterval(() => {
  const now = Date.now();
  for (const [id, session] of sessions) {
    if (now > session.expiresAt) sessions.delete(id);
  }
}, 5 * 60 * 1000).unref(); // .unref() so it doesn't block process exit
```

**Effort:** 10 minutes
**Regression risk:** None

### 13.19 — Annotation extraction and selector lookup duplicated across 4 files

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Systemic |
| **Files** | `server/src/tools/analyze-patterns.js:33-40`, `generate-spec.js:37-44`, `detect-recurring-issues.js:35-42`, `server/src/analysis/annotation-diff.js:31`, `recurring-issues.js:24`, `state-machine.js:26`, `steering-generator.js:37` |

```js
for (const { parsed } of results) {
  if (!parsed.annotations?.length) continue;

```

**Pattern B — selector extraction:**
```js
ann.ancestor || ann.element?.selector || null
```
This fallback chain appears verbatim in `annotation-diff.js`, `recurring-issues.js`, `state-machine.js`, and `steering-generator.js`. `ancestor` and `element.selector` are semantically different fields and the silent fallback may use the wrong one in edge cases.

```js
export function extractAllAnnotations(results) {

}
export function getAnnotationSelector(ann) {
  return ann.selector ?? ann.element?.selector ?? ann.ancestor ?? null;
}
```

**Effort:** 1 hour
**Regression risk:** Low — behavior identical if extraction is correct

### 13.22 — No logger abstraction: 15 `console.log` calls vs 3 `process.stderr.write` calls

| Attribute | Value |
|---|---|
| **Severity** | LOW |
| **Scope** | Systemic |
| **Files** | Across `server/src/` |

### 13.26 — `get-capture.js` has unguarded async file I/O in the diff-export path

| Attribute | Value |
|---|---|
| **Severity** | LOW |
| **Scope** | Local |
| **Files** | `server/src/tools/get-capture.js:59,79,90` |

**Evidence:** The diff export path dynamically imports `fs`, calls `mkdirSync`, and calls `writeFileSync` without a wrapping try-catch. A permission error or full disk would produce an unhandled rejection and crash the tool handler instead of returning a clean error response.

**Recommended fix:**
```js
try {
  const { writeFileSync, mkdirSync } = await import('fs');
  mkdirSync(path.dirname(outputPath), { recursive: true });
  writeFileSync(outputPath, result.diffBuffer);
} catch (err) {
  return errorResponse(`Failed to write diff file: ${err.message}`);
}
```

**Effort:** 10 minutes
**Regression risk:** None

## Summary

The server's core infrastructure (auth, path validation, indexer, watcher) is solid. The performance and silent-failure problems are concentrated in the analysis pipeline and the multi-file tool layer. Phase 1 takes 90 minutes and has zero regression risk. Phase 2 is a one-day sprint with straightforward changes. Phase 3 (consistency-checker O(N²), withCapture wrapper, http-receiver split) should be its own code-health milestone before the v1.0 launch prep — these are the items most likely to cause user-visible degradation as capture counts grow.
