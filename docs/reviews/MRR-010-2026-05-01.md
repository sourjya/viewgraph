3. **Hardcoded hex colors grew to 117** across 18 extension files (was 102 in MRR-009, 84 in MRR-004). The trend is worsening with each sidebar feature addition.

**New positive developments:**

- Gateway clustering is well-designed and well-tested (8 test cases covering all edge cases)
- `shared-params.js` adopted by 10 tools, eliminating duplicate Zod schema definitions
- CI workflow covers server tests, extension tests, vitest plugin tests, extension build, and bundle size gate
- Extension perf pipeline (bundle size budgets, benchmarks, Puppeteer E2E) is a solid foundation

## New Findings

### 15.1 — Vitest plugin `captureDOM()` writes to disk by default despite JSDoc claiming `write: false`

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Local |
| **Files** | `packages/vitest/index.js:172` |

**Evidence:**

JSDoc at line 128 says: `@param {boolean} [options.write] - Write to disk (default: false)`

But the code at line 172:
```js
if (options.write !== false) {
```

When `options.write` is `undefined` (the default), `undefined !== false` evaluates to `true`, so the function writes to disk. Every `captureDOM('label')` call without explicit `{ write: false }` silently creates files in `.viewgraph/captures/`.

The tests all pass `{ write: false }` explicitly, masking the bug. The `captureAndAssert` function correctly passes `{ write: false, ...options }` as its default, but direct `captureDOM` callers are affected.

The TypeScript types in `index.d.ts` declare `write?: boolean` with no default annotation, which is ambiguous.

**Recommended fix:**
```js
if (options.write === true) {  // explicit opt-in
```
Or update JSDoc to document the actual behavior (write-by-default).

**Effort:** 2 minutes
**Regression risk:** Low — changes default behavior, but the current default is the bug

### 15.2 — `readAndParseMulti()` attaches `.warnings` property to an Array — non-standard mutation

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Files** | `server/src/utils/tool-helpers.js:178` |

**Evidence:**
```js
const results = settled.filter(Boolean);
const warnings = filenames.length - results.length > 0
  ? [`${filenames.length - results.length} of ${filenames.length} captures could not be read`]
  : [];
results.warnings = warnings;
return results;
```

This attaches a `.warnings` property to a plain Array. While JavaScript allows this, it's a code smell:
- The return type annotation says `Promise<Array<...>>` — callers don't expect extra properties
- `JSON.stringify(results)` silently drops `.warnings`
- TypeScript would flag this as an error
- No caller currently reads `.warnings` — it's dead data

**Recommended fix:** Return a proper object:
```js
return { results: settled.filter(Boolean), warnings };
```

This requires updating 7 callers (analyze-journey, analyze-patterns, detect-recurring, diff-annotations, generate-spec, list-sessions, visualize-flow) to destructure `{ results }` instead of using the array directly.

**Effort:** 30 minutes
**Regression risk:** Low — mechanical change, all callers are known

### 15.7 — Gateway `safeOptional()` falls back to `z.any()` for ZodEffects — loses type safety

| Attribute | Value |
|---|---|
| **Severity** | LOW |
| **Scope** | Local |
| **Files** | `server/src/clusters/gateway.js:50-53` |

**Evidence:**
```js
if (typeName === 'ZodEffects') {
  return z.any().optional();
}
```

When a sub-tool parameter has a Zod transform or refinement (ZodEffects), the gateway replaces it with `z.any().optional()`. This means the gateway schema accepts any value for that parameter — the sub-tool handler still validates, but the gateway's merged schema is weaker than it should be.

Currently no tools use ZodEffects parameters, so this is defensive code with no active impact. But if a tool adds `.transform()` or `.refine()` to a parameter, the gateway silently drops type checking for it.

**Recommended fix:** Add a comment documenting this limitation. If ZodEffects params appear in the future, consider unwrapping the inner type instead of falling back to `z.any()`.

**Effort:** 5 minutes (comment), 30 minutes (unwrap implementation)
**Regression risk:** None

### 15.8 — Vitest plugin capture format diverges from Playwright/extension format

| Attribute | Value |
|---|---|
| **Severity** | LOW |
| **Scope** | Local |
| **Files** | `packages/vitest/index.js` (capture structure), `packages/vitest/index.d.ts` (types) |

**Evidence:** The Vitest plugin produces a simplified capture format:
- `nodes` is a flat array of `{ nid, tag, role, text }` — missing `alias`, `component`, `actions`, `bbox`, `visibleText`
- `actionManifest.byAction` entries lack `ref`, `axName`, `locator`, `inViewport`
- No `summary`, `details`, `relations` sections
- No `_notice` prompt injection defense markers

The Playwright plugin and extension produce the full v2.4.0 format. MCP server tools that expect the full format (e.g., `get-component-coverage`, `audit-accessibility`) will produce incomplete results when given a Vitest capture.

This is intentional — jsdom doesn't have layout, computed styles, or bounding boxes. But the format divergence should be documented in the capture's `metadata` so server tools can detect it and adjust.

**Recommended fix:** Add `metadata.capabilities: ['structure', 'roles', 'testids']` (vs `['structure', 'roles', 'testids', 'layout', 'styles', 'bbox']` for extension captures). Server tools can check capabilities before attempting layout/style analysis.

**Effort:** 1 hour
**Regression risk:** None — additive metadata

## Focus Area Assessments

### 1. Gateway Proxy Complexity — Maintainable ✅

The gateway implementation in `gateway.js` (225 lines) is well-structured:

- **Clean separation:** `createToolProxy()` handles registration interception, `registerGateways()` handles gateway creation, `buildDiscoveryText()` handles help text. Three functions, three responsibilities.
- **Edge cases handled:** Single-tool clusters use `z.literal()` instead of `z.enum()` (Fix 5). Already-optional params aren't double-wrapped (Fix 2). ZodEffects fall back to `z.any()` (defensive). MCP context is forwarded (Fix 4).
- **Config-driven:** `cluster-config.json` is the single source of truth for tool groupings. Adding/removing tools from clusters requires zero code changes.
- **Well-tested:** 8 test cases in `gateway.test.js` cover flat mode, clustered mode, discovery, dispatch, context forwarding, unknown actions, single-tool clusters, and empty clusters.
- **Mode switching in `index.js`** is clean: 5 lines of conditional logic at lines 110-111 and 180-183. The proxy pattern means all 41 `register*()` calls are identical regardless of mode.

**One concern:** The `safeOptional()` fallback to `z.any()` for ZodEffects (finding 15.7) is a latent type safety gap. Currently harmless but worth documenting.

### 2. Vitest Plugin Code Quality — Good with One Bug

The plugin is intentionally minimal (208 lines, 2 exports, 9 tests). Code quality is solid:

- Clean JSDoc on all public functions
- Proper TypeScript types in `index.d.ts`

**Issues:**

## Positive Observations

1. **withCapture() migration is effectively complete.** From 1/12 to 10/12 in one cycle. The 2 remaining inline uses are justified by multi-step logic. This eliminates the #1 duplication pattern from MRR-008/009.
