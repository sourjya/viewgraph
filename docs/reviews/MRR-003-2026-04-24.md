# MRR-003 - Maintainability Review Report (Delta)

**Date:** 2026-04-24
**Baseline:** MRR-001 (2026-04-21)
**Scope:** New auth modules, extension auth, annotate.js marker lifecycle, sync.js, transport.js, settings.js, help.js
**Cross-reference:** SRR-005 (2026-04-24) - security review of same changeset

---

## Executive Summary

This delta covers the HMAC auth system (ADR-015) added across 5 new files (4 server, 1 extension) plus UX changes for resolved marker visibility. The auth modules are cleanly separated with good single-responsibility boundaries. However, the implementation has **critical security flaws** (see SRR-005) that must be fixed before the auth code provides any value. From a maintainability perspective, the new code is well-structured but has one significant gap: **zero test coverage for extension auth.js** (103 lines of crypto code with no tests).

**New code:** ~550 LOC across 5 new files + ~80 LOC changes to 4 existing files
**New tests:** 30 tests across 4 server auth test files. 0 extension auth tests.
**Test totals:** 522 server + 1143 extension = 1665 total (up from ~1562 at MRR-001)

## 1. New Auth Modules - server/src/auth/*.js

### 1.1 Module Separation (GOOD)

4 files with clean single-responsibility:

| File | Lines | Responsibility | Dependencies |
|---|---|---|---|
| `hmac.js` | 42 | Sign, verify, hashBody - pure crypto | `crypto` only |
| `session-key.js` | 30 | Generate + read `.session-key` file | `crypto`, `fs` |
| `session-store.js` | 42 | In-memory session Map with TTL | `crypto` (randomUUID) |
| `middleware.js` | 100 | Handshake flow + request validation | hmac.js, session-store.js |

The dependency graph is a clean DAG: `middleware → hmac + session-store`, `session-key` is standalone. No circular dependencies. Each module is independently testable.

### 1.2 hmac.js (GOOD)

- Uses `timingSafeEqual` for signature verification - correct
- `hashBody` handles null/empty gracefully
- Pure functions, no state, easy to test
- JSDoc on all exports

### 1.3 session-key.js (GOOD)

- `chmodSync(filePath, 0o600)` - correct file permissions
- `readSessionKey` returns null on missing file (no throw) - clean error contract
- Synchronous I/O is acceptable here (runs once at startup)

### 1.4 session-store.js (GOOD)

- Factory pattern (`createSessionStore()`) - testable, no singletons
- TTL-based expiry with lazy cleanup on `validate()`
- **Minor:** No proactive cleanup of expired sessions (SRR-005 S4-6, LOW). Acceptable for beta - sessions are ~100 bytes each, server restarts clear everything.

### 1.5 middleware.js (MIXED - security issues)

**Good:**
- Clean separation of `handleHandshake`, `handleVerify`, `validateRequest`
- Exempt routes defined as a Set (O(1) lookup)
- Timestamp window validation (30s)
- Challenge TTL (60s) with cleanup

**Issues (all flagged in SRR-005):**
- **S4-2 CRITICAL:** `handleHandshake()` returns `{ challenge, key: secret }` - leaks the HMAC secret in the HTTP response. One-line fix: remove `key: secret`.
- **S4-4 HIGH:** `handleVerify()` line 63 uses `===` instead of `timingSafeEqual` for challenge-response comparison. Inconsistent with `hmac.js` which correctly uses `timingSafeEqual`.
- **S4-7 LOW:** `pendingChallenges` Map is unbounded. Add a cap (e.g., 100).

### 1.6 http-receiver.js integration (MIXED)

- **S4-3 HIGH:** Line 161 passes `body: ''` (hardcoded) to `validateRequest()`. The extension correctly signs the body hash, but the server ignores it. Body must be read before auth validation.
- **S5-8 MEDIUM:** Line 150 `JSON.parse(body || '{}')` - no try/catch on `/handshake/verify`. Inconsistent with other POST handlers that wrap JSON.parse.

## 2. Extension auth.js (NEEDS WORK)

### 2.1 Web Crypto Usage (GOOD)

- Uses `crypto.subtle.importKey` + `crypto.subtle.sign` - correct browser API
- `hexToBytes` / `bytesToHex` are clean manual implementations
- `sha256` uses `crypto.subtle.digest` - correct
- Graceful fallback: `if (!crypto?.subtle) return ''` - handles test environments

### 2.2 Error Handling (GOOD)

- `authenticate()` wraps everything in try/catch, returns `false` on failure
- `AbortSignal.timeout(3000)` on both fetch calls - prevents hanging
- `clearAuth()` resets all state cleanly

### 2.3 Module State (ACCEPTABLE)

- Module-level `_sessionId`, `_secret`, `_authenticated` - standard pattern for singleton auth state
- `clearAuth()` provides clean teardown
- No risk of stale state across page navigations (sidebar destroy calls `clearAuth`)

### 2.4 Security Design Flaw (CRITICAL - mirrors S4-2)

Line 30: `const { challenge, key } = await res.json()` - the extension **expects** the server to send the secret in the handshake response. This is the client-side counterpart of S4-2. When S4-2 is fixed (secret removed from response), the extension must be updated to read the key from `.session-key` via background script or native messaging.

### 2.5 Test Coverage (GAP - HIGH)

**Zero test files for extension auth.js.** This is 103 lines of crypto code with:
- HMAC signing logic that must match the server's `hmac.js`
- SHA-256 hashing that must produce identical output to Node's `createHash`
- Hex encoding/decoding that must be byte-perfect
- State management (authenticate, clearAuth, isAuthenticated)

The `crypto.subtle` fallback (`return ''`) means tests can run in jsdom, but no tests exist. This is the highest-priority test gap in this changeset.

## 3. annotate.js Changes - Marker Lifecycle

### 3.1 dimMarker / dimResolvedMarker (GOOD)

- `dimMarker(id)` reduces opacity to 0.3, grays border, replaces badge text with '✓'
- Only modifies `style` properties and `textContent` - no XSS risk (confirmed in SRR-005 I-3)
- `dimResolvedMarker` is a thin export wrapper - clean API boundary

### 3.2 updateResolvedMarkerVisibility (GOOD)

- Iterates `annotations` array, toggles `display` based on filter tab
- Correctly handles the three filter states: 'open' (hide resolved), 'resolved'/'all' (show)
- Called from sidebar tab switching - clean separation of concerns

### 3.3 spotlightResolvedMarker (GOOD)

- Temporarily raises opacity to 0.7 for 2 seconds, then fades back to 0.3
- Uses CSS transition for smooth animation
- No memory leak risk - single `setTimeout` with no cleanup needed

### 3.4 Resolved marker hidden on load (GOOD)

In `load()`, resolved annotations get `dimMarker()` + `display: 'none'`. This prevents a flash of unresolved markers on page reload. Users see resolved markers only when switching to the Resolved/All tab.

### 3.5 resolveAnnotation one-way (GOOD)

`resolveAnnotation()` replaces the old `toggleResolved()` (kept as deprecated alias). One-way resolve is the correct UX - resolved items should not be un-resolved from the extension.

## 4. sync.js Changes

### 4.1 Dynamic Import for dimResolvedMarker (ACCEPTABLE)

```js
const { dimResolvedMarker } = await import('#lib/annotate.js');
```

This avoids a circular dependency: `sync.js` imports `annotate.js` at the top for `getAnnotations()`, but `dimResolvedMarker` is only needed inside the async `syncResolved` callback. The dynamic import is a pragmatic solution.

**Minor concern:** Two imports from the same module - one static (`getAnnotations`), one dynamic (`dimResolvedMarker`, `save`). This works but is slightly unusual. A cleaner alternative would be to import both statically since there's no actual circular dependency risk here (sync.js doesn't export anything that annotate.js imports).

### 4.2 Save After Resolve (GOOD)

```js
const { save } = await import('#lib/annotate.js');
await save();
```

Persists resolved state immediately after updating annotations. Prevents the race condition where the sidebar could close before resolved state is written to `chrome.storage.local`. Confirmed safe in SRR-005 I-4.

## 5. transport.js Changes

### 5.1 Lazy Auth Import (GOOD)

```js
let _authModule = null;
async function getAuth() {
  if (!_authModule) {
    try { _authModule = await import('./auth.js'); }
    catch { _authModule = { signRequest: async () => ({}), isAuthenticated: () => false }; }
  }
  return _authModule;
}
```

- Avoids `crypto.subtle` errors in jsdom test environment
- Fallback stub returns empty headers and `isAuthenticated: false` - safe defaults
- Cached after first load - no repeated dynamic imports
- Confirmed safe in SRR-005 I-6

### 5.2 getServerUrl Export (GOOD)

Simple getter: `export function getServerUrl() { return _serverUrl; }`. Used by `settings.js` to extract the port number for the version info display. No sensitive data exposed.

### 5.3 Auth Header Integration (GOOD)

Both `_query` and `_send` conditionally add auth headers:
```js
const authHeaders = (await getAuth()).isAuthenticated()
  ? await (await getAuth()).signRequest(method, path, body)
  : {};
```

Clean conditional - no auth headers sent when not authenticated. The double `await getAuth()` is slightly redundant (could cache the result), but the module-level cache in `getAuth()` makes this a no-op after first call.

## 6. settings.js - Version Pills

### 6.1 Version Info Display (GOOD)

Renders three pills: `Ext v0.4.x`, `Server v0.4.x`, `Port 9876`. Uses `chrome.runtime.getManifest().version` for extension version and server `/info` response for server version. Clean presentation.

### 6.2 svgFromString Usage (ACCEPTABLE)

```js
helpLink.replaceChildren(svgFromString('<svg ...>...</svg>'), document.createTextNode('URL mapping docs'));
```

Single call site with a hardcoded SVG string. `svgFromString` uses `DOMParser` with `image/svg+xml` which does not execute scripts. Confirmed safe in SRR-005 I-2.

**Minor:** The `advLink` on line ~120 still uses `innerHTML` for its SVG icon instead of `svgFromString`. Inconsistent - one link uses the safe helper, the adjacent link uses innerHTML. Both are safe (hardcoded strings), but the inconsistency should be cleaned up.

### 6.3 settings.js innerHTML (MEDIUM)

3 `innerHTML` assignments remain in settings.js (advLink icon, advLink toggle icon, all-servers collapse icon). All use hardcoded SVG strings. Should migrate to `svgFromString` or `icons.js` factories for consistency with the project's policy.

## 10. New Findings

### 10.1 Extension auth.js needs tests (HIGH - NEW)

103 lines of crypto code with zero tests. Highest-priority test gap. See section 8.2 for specific test cases needed.

### 10.2 Auth design requires rework before providing value (HIGH - NEW, cross-ref SRR-005)

The HMAC auth system has 1 CRITICAL + 2 HIGH security issues. Until S4-2 is fixed (secret in handshake response), the auth system provides zero security benefit. The extension's `authenticate()` function must be redesigned to read the key from disk (via background script or native messaging) rather than from the HTTP response.

**Maintainability impact:** When S4-2 is fixed, both `middleware.js` (remove `key: secret`) and `extension/lib/auth.js` (new key acquisition path) must change in lockstep. Document this coupling.

### 10.3 sync.js mixed static/dynamic imports from same module (LOW - NEW)

`sync.js` imports `getAnnotations` statically from `annotate.js` but imports `dimResolvedMarker` and `save` dynamically. No circular dependency exists - both could be static imports. The dynamic import adds unnecessary complexity.

### 10.4 settings.js innerHTML inconsistency (LOW - NEW)

`helpLink` uses `svgFromString()` but adjacent `advLink` uses `innerHTML`. Both are safe (hardcoded SVGs) but the inconsistency undermines the project's innerHTML elimination effort. 3 innerHTML assignments remain in settings.js.
