# MRR-004 - Maintainability Review Report (Full Codebase)

**Date:** 2026-04-27
**Baseline:** MRR-003 (2026-04-24)
**Scope:** Full codebase - all source files in `server/src/`, `extension/lib/`, `extension/entrypoints/`
**Cross-reference:** SRR-006 (2026-04-27) - security review of same codebase

---

## Executive Summary

This is a full-codebase maintainability review covering 10,846 LOC across 62 extension files and 6,644 LOC across 59 server files. The codebase has improved significantly since MRR-001 (sidebar decomposition, readAndParse migration, COLOR constants). However, three systemic issues dominate:

1. **Duplicate discovery+status block in annotation-sidebar.js** - The `discoverServer().then()` block (lines 103-195 and 259-355) is copy-pasted verbatim. This is ~100 lines of duplicated async logic including trust classification, version checking, offline banners, and auth mode detection.

2. **84 hardcoded hex colors** across sidebar modules and annotation-panel.js, despite `COLOR` constants existing in `styles.js`. This was flagged as 76 in MRR-001 and has *regressed* to 84.

3. **17 server tools bypass `readAndParse()`** and do their own `readFile` + `validateCapturePath` + `parseCapture` inline. This was flagged as 16 in MRR-001 and has grown to 17 with `get-unresolved.js`.

**New findings:** 14 (4 HIGH, 6 MEDIUM, 4 LOW)
**Carried findings from MRR-001/003:** 13 open
**Resolved since MRR-003:** 0

**Codebase totals:** ~17,490 LOC source, 143 test files, 1,646 tests

## New Findings

### 11.1 - Duplicate discoverServer().then() block in annotation-sidebar.js

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Local |
| **Files** | `extension/lib/annotation-sidebar.js:103-195` and `:259-355` |

**Evidence:** Two nearly identical `discoverServer(window.location.href).then(async (url) => { ... })` blocks in `create()`. Both perform:
- Server discovery and status dot coloring
- Version mismatch banner construction (identical DOM creation code)
- Trust classification via `classifyTrust()`
- Offline mode detection with "no matching project" vs "server timed out" banners
- Auth mode detection (native vs HMAC)

The second block (line 259) was introduced for "Version info for help card" but duplicates the entire first block. The only difference: the second block adds native messaging detection (`transport.isNative()`).

**Impact:** Any bug fix to the discovery/status logic must be applied in two places. The second block already has a stale comment ("Version info now shown in Settings panel, not help card") suggesting it was copy-pasted and partially edited.

**Recommended refactor:** Extract a `async function initServerConnection()` that runs once, handles all discovery, trust, version, and auth logic. Call it once from `create()`.

**Effort:** 1-2 hours
**Regression risk:** Medium - must test all status banner states (connected, no match, timed out)

### 11.3 - `pushSnapshot()` and `pushScreenshot()` use hardcoded `SERVER_URL`

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Files** | `extension/entrypoints/background.js:140,162` |

**Evidence:** Both functions use `SERVER_URL` (the static fallback `http://127.0.0.1:9876`) instead of the dynamically discovered server URL. The main capture push at line 111 correctly uses `await discoverServer(pageUrl, capturesDir) || SERVER_URL`, but snapshots and screenshots bypass discovery entirely.

In multi-project setups, this sends snapshots/screenshots to the wrong server (always port 9876 regardless of which project the page belongs to).

**Recommended refactor:** Pass the discovered `serverUrl` to `pushSnapshot()` and `pushScreenshot()` as a parameter.

**Effort:** 15 minutes
**Regression risk:** Low

### 11.8 - 17 server tools bypass readAndParse()

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Systemic |
| **Files** | `check-consistency.js`, `compare-baseline.js`, `compare-screenshots.js`, `find-source.js`, `get-capture-stats.js`, `get-capture.js`, `get-latest.js`, `get-session.js`, `get-unresolved.js`, `resolve-annotation.js`, `get-request-status.js`, `get-session-status.js`, `list-archived.js`, `list-baselines.js`, `list-captures.js`, `request-capture.js`, `set-baseline.js` |

**Evidence:** These tools do their own `readFile` + `validateCapturePath` + `parseCapture` instead of using the centralized `readAndParse()` helper from `tool-helpers.js`. This means:
- Fuzzy filename matching (suggestFilename) is not available
- Error messages are inconsistent ("Capture not found" vs "Error: Invalid filename")
- The "did you mean?" suggestion feature only works on 21/38 tools

Some tools have legitimate reasons (e.g., `get-capture.js` needs archive fallback, `list-captures.js` doesn't read files). But at least 10 could be migrated.

**Recommended refactor:** Migrate `check-consistency.js`, `compare-baseline.js`, `find-source.js`, `get-capture-stats.js`, `get-latest.js`, `get-session.js`, `get-unresolved.js`, `resolve-annotation.js`, `compare-screenshots.js` to use `readAndParse()`. For tools needing special behavior (archive fallback, multi-file), extend `readAndParse` with options.

**Effort:** 3-4 hours
**Regression risk:** Medium - each tool needs its test verified

### 11.10 - 7 register() signature variations across 38 tools

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Systemic |
| **Files** | All 38 files in `server/src/tools/` |

**Evidence:** Tool registration has 7 distinct signatures:
```
20 tools: register(server, _indexer, capturesDir)
10 tools: register(server, indexer, capturesDir)
 2 tools: register(server, queue)
 2 tools: register(server, indexer)
 2 tools: register(server, _indexer, capturesDir, options = {})
 1 tool:  register(server, indexer, capturesDir, options = {})
 1 tool:  register(server, capturesDir)
```

The `_indexer` vs `indexer` naming is inconsistent (20 tools prefix with underscore to suppress unused-var warnings, 10 don't). The `options` parameter is used by 3 tools for WebSocket broadcast callbacks.

**Recommended refactor:** Standardize on a single signature: `register(server, context)` where `context = { indexer, capturesDir, queue, broadcast }`. This eliminates positional argument confusion and makes adding new shared dependencies trivial.

**Effort:** 4-6 hours (38 files + index.js + all test files)
**Regression risk:** High - touches every tool file. Do as a dedicated refactor branch.

### 11.11 - 76 lines of import+register boilerplate in server/index.js

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Files** | `server/index.js:28-147` |

**Evidence:** 38 individual `import { register as registerXxx }` lines followed by 38 `registerXxx(server, ...)` calls. Every new tool requires adding 2 lines here. This was flagged in the modularity audit as a candidate for auto-discovery.

**Recommended refactor:** Auto-discovery pattern:
```js
const toolFiles = await glob('src/tools/*.js', { cwd: __dirname });
for (const file of toolFiles) {
  const { register } = await import(file);
  register(server, context);
}
```
Requires the unified `context` object from finding 11.10.

**Effort:** 1 hour (after 11.10 is done)
**Regression risk:** Medium - must verify all tools still register correctly

## Positive Observations

1. **Sidebar decomposition is working.** `annotation-sidebar.js` is down from 2,268 lines (MRR-001) to 830 lines. The extracted modules (`review.js`, `settings.js`, `diagnostics.js`, `captures.js`, etc.) have clear boundaries.

2. **Event bus architecture is clean.** `sidebar/events.js` defines all inter-module events. No sidebar sub-module imports another directly. This was a key recommendation from the modularity audit.

3. **`readAndParse` adoption is growing.** 21 of 38 tools now use it, up from 14 at MRR-001. The fuzzy filename matching it provides is a real UX improvement.

4. **New modules follow conventions.** `archive.js`, `list-archived.js`, `get-unresolved.js`, `get-annotation-context.js` all follow established patterns and have test files.

5. **`annotation-types.js` type registry is well-designed.** Centralizes type resolution, badge colors, severity rules. Exactly the pattern recommended in the modularity audit.

## Summary

The codebase is growing faster than its test coverage and abstraction layer. The sidebar decomposition was a major win, but new features are being added with hardcoded colors, bypassing readAndParse, and without dedicated tests. The duplicate discoverServer block (11.1) is the most urgent fix - it's a maintenance trap that will cause bugs on the next status banner change.

Phase 1 items are all safe to do immediately. Phase 2 items should be batched into a "code health" sprint before the next feature milestone. Phase 3 items (unified tool context + auto-discovery) would be the single highest-leverage structural improvement for the server codebase.
