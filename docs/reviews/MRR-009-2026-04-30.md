## Executive Summary

MRR-008 was produced earlier today. This review tracks what was fixed, what remains open, and identifies 8 new findings. The good news: 5 of MRR-008's 22 findings have been resolved (13.1, 13.5, 13.6, 13.17, 13.18). The `withCapture()` wrapper now exists and `readAndParseMulti` is parallel. However, adoption of `withCapture()` stalled at 1 of 12 eligible tools — the wrapper exists but wasn't rolled out.

Three systemic problems persist:

1. **Incomplete migration to shared abstractions.** `withCapture()` is used by exactly 1 tool (`audit-accessibility.js`). The remaining 11 single-capture tools still inline the `readAndParse` + error-check pattern. Similarly, `jsonResponse()`/`errorResponse()` adoption is incomplete — tool response shapes remain inconsistent.

2. **Extension color sprawl worsening.** Hardcoded hex colors grew from 84 (MRR-004) to 102 across 19 extension files. No CSS custom properties or theme object was introduced.

3. **verify-fix.js has a latent bug** from the `readAndParse` migration — it checks `!result.parsed` instead of `!result.ok`, which works by accident but loses the helpful error message (filename suggestions, available captures list).

**New findings:** 8 (1 HIGH, 4 MEDIUM, 3 LOW)
**Resolved from MRR-008:** 5 (13.1, 13.5, 13.6, 13.17, 13.18 partially)
**Still open from MRR-008:** 17

## New Findings

### 14.1 — `verify-fix.js` checks `!result.parsed` instead of `!result.ok` — loses error context

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Local |
| **Files** | `server/src/tools/verify-fix.js:175-176` |

**Evidence:**
```js
const result = await readAndParse(targetFile, capturesDir);
if (!result.parsed) return errorResponse(`Cannot read capture: ${targetFile}`);
```

`readAndParse` returns `{ ok: false, error: <MCP response> }` on failure. There is no `.parsed` property on the failure path, so `!result.parsed` is truthy and the check works — but it discards the rich error from `readAndParse` (which includes "Did you mean: ..." suggestions and available captures list) and replaces it with a generic "Cannot read capture" message. Every other tool in the codebase uses `if (!ok) return error;`.

**Recommended fix:**
```js
const { ok, parsed, error } = await readAndParse(targetFile, capturesDir);
if (!ok) return error;
```

**Effort:** 2 minutes
**Regression risk:** None — improves error messages

### 14.2 — `withCapture()` adopted by only 1 of 12 eligible tools — migration stalled

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Systemic |
| **Files** | `server/src/tools/audit-accessibility.js` (uses it), 11 others don't: `audit-layout.js`, `find-missing-testids.js`, `get-annotations.js`, `get-annotation-context.js`, `get-component-coverage.js`, `get-elements-by-role.js`, `get-fidelity-report.js`, `get-interactive.js`, `get-page-summary.js`, `validate-capture.js`, `verify-fix.js` |

**Evidence:** All 11 tools still use the inline pattern:
```js
const { ok, parsed, error } = await readAndParse(filename, capturesDir);
if (!ok) return error;
```

The `withCapture()` wrapper at `tool-helpers.js:230-240` was created to eliminate this exact duplication (MRR-008 finding 13.18). Only `audit-accessibility.js` was migrated. The remaining 11 tools are straightforward candidates — they take a single filename, parse it, and operate on the result.

**Recommended fix:** Migrate each tool. Example for `audit-layout.js`:
```js
// Before (4 lines)
const { ok, parsed, error } = await readAndParse(filename, capturesDir);
if (!ok) return error;
const layout = analyzeLayout(parsed);
return jsonResponse(layout);

// After (1 line + callback)
return withCapture(filename, capturesDir, (parsed) => {
  return jsonResponse(analyzeLayout(parsed));
});
```

**Effort:** 1-2 hours for all 11 tools
**Regression risk:** Low — each migration is mechanical

### 14.4 — `console.error` count on stderr grew from 15 to 18 — no logger abstraction

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Systemic |
| **Files** | `server/src/` (18 occurrences across 9 files) |

**Evidence:** MRR-008 finding 13.22 counted 15 `console.log`/`console.error` calls and recommended a `log()` helper. The count is now 18 — new code in `post-capture-audit.js` and `tool-helpers.js` added `console.error` calls directly. All use `console.error` (not `console.log`), so the MCP stdout corruption risk from MRR-008 is mitigated, but the lack of a centralized logger means:
- No structured log format
- No log level filtering
- The `LOG_PREFIX` constant is manually prepended in each call

**Recommended fix:** Add a 5-line logger module:
```js
// server/src/logger.js
import { LOG_PREFIX } from './constants.js';
export const log = {
  info: (msg) => process.stderr.write(`${LOG_PREFIX} ${msg}\n`),
  error: (msg) => process.stderr.write(`${LOG_PREFIX} ERROR: ${msg}\n`),
  warn: (msg) => process.stderr.write(`${LOG_PREFIX} WARN: ${msg}\n`),
};
```

**Effort:** 30 minutes
**Regression risk:** None

### 14.5 — `get-capture-diff.js` still has a size guard that prevents diffing captures > 1MB each

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Files** | `server/src/tools/get-capture-diff.js:82-88` |

**Evidence:**
```js
if (prevStr.length + targetStr.length > 2_000_000) {
  return jsonResponse({ mode: 'full', reason: `Captures too large for diff...` });
}
```

This guard was added as a safety measure, but it means any page with >1000 elements (which produces captures >1MB) silently falls back to "full" mode — defeating the purpose of the diff tool. The `fast-json-patch` library handles large objects fine; the real concern is the patch size, which is already checked at line 100 (50% threshold). The pre-check is redundant and overly conservative.

**Recommended fix:** Remove the 2MB pre-check or raise it to 10MB. The 50% patch-size check at line 100 is the correct guard.

**Effort:** 5 minutes
**Regression risk:** None — removes a false-negative guard

### 14.8 — Tool parameter naming uses `snake_case` for MCP schema but `camelCase` internally — inconsistent destructuring

| Attribute | Value |
|---|---|
| **Severity** | LOW |
| **Scope** | Systemic |
| **Files** | `server/src/tools/compare-styles.js:32`, `compare-captures.js:30`, `compare-screenshots.js:40`, `list-captures.js:30`, `get-latest.js:35`, `list-archived.js:34`, `get-capture-history.js:24`, `get-annotation-context.js:33` |

**Evidence:** MCP tool parameters use `snake_case` (e.g., `file_a`, `file_b`, `url_filter`, `element_id`, `annotation_id`) because the MCP protocol convention is snake_case. However, the destructuring in tool handlers is inconsistent:

```js
// Some tools rename on destructure (good):
async ({ limit, url_filter: urlFilter }) => { ... }

// Others use snake_case throughout the handler (inconsistent with JS convention):
async ({ file_a, file_b, element_id }) => { ... }
```

8 tools use snake_case variables internally. This is a style inconsistency, not a bug, but it creates confusion about whether the codebase convention is camelCase (it is everywhere else).

**Recommended fix:** Rename on destructure in all 8 tools:
```js
async ({ file_a: fileA, file_b: fileB, element_id: elementId }) => { ... }
```

**Effort:** 30 minutes
**Regression risk:** None — internal variable names only
