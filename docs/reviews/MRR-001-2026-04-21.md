# MRR-001 - Maintainability Review Report

**Date:** 2026-04-21
**Scope:** extension/lib/sidebar/*.js, extension/lib/collectors/*.js, server/src/tools/*.js, server/src/*.js
**Codebase:** ~10,116 LOC extension/lib, ~5,958 LOC server/src, 37 MCP tools, 18 sidebar modules, 19 collectors
**Previous reviews:** Codebase Review (2026-04-18), SRR-004 (2026-04-21)

---

## Executive Summary

The codebase is well-structured for its age and velocity. The sidebar decomposition (F14) was successful - annotation-sidebar.js is down to 647 lines/14 functions from the original 2,268. Collectors have a shared `collector-utils.js`. The event bus prevents circular dependencies between sidebar modules.

The three highest-impact maintainability risks are:
1. **16 server tools still bypass `readAndParse`** - missing fuzzy filename suggestions, inconsistent error messages, ~200 lines of duplicated validation
2. **76 hardcoded CSS colors in sidebar** despite `styles.js` COLOR constants existing and being imported by all 13 sidebar files
3. **16 server tools have no dedicated test files** - nearly half the tool surface area

---

## 1. Code Duplication

### 1.1 Server tool file-reading fragmentation (HIGH)

**Evidence:** 21 of 37 tools use `readAndParse` from `tool-helpers.js`. The remaining 16 manually import `readFile` + `validateCapturePath` and reimplement the read-parse-error cycle:
- `get-capture.js`, `get-latest.js`, `compare-baseline.js`, `check-consistency.js`, `find-source.js`, `get-session.js`, `get-capture-stats.js`, `compare-screenshots.js`, `get-unresolved.js`, `list-baselines.js`, `list-captures.js`, `resolve-annotation.js`, `set-baseline.js`, `get-request-status.js`, `get-session-status.js`, `request-capture.js`

**Impact:** No fuzzy filename suggestions on these 16 tools. Inconsistent error messages. ~200 duplicated lines.

### 1.2 Hardcoded CSS colors in sidebar (HIGH)

**Evidence:** 76 hardcoded color strings across sidebar files. `styles.js` exports a `COLOR` object with semantic names (`muted`, `secondary`, `success`, `warning`, `error`, `primary`), and all 13 sidebar files import from `styles.js` - but most still use raw hex values inline.

Top offenders: `#60a5fa` (7x), `#666` (6x), `#1e3a5f` (6x), `#4ade80` (4x), `#9ca3af` (3x).

`icons.js` defaults to `'#666'` in 4 function signatures instead of `COLOR.muted`.

### 1.3 Hover event listener pairs (MEDIUM)

**Evidence:** 40+ `mouseenter`/`mouseleave` pairs across 8 sidebar files (footer: 8, header: 8, review: 6, suggestions-ui: 4, settings: 4, help: 4, inspect: 4, strip: 2). Each pair is 4-6 lines of identical pattern.

### 1.4 Server tool response formatting (LOW - partially addressed)

**Evidence:** `jsonResponse()` and `errorResponse()` helpers exist in `tool-helpers.js` and are used by 35 and 19 tools respectively. Only 2 tools still use raw `JSON.stringify` in response construction. This was a major issue in the 2026-04-18 review and has been largely fixed.

## 2. Module Coupling

### 2.1 No circular dependencies detected (PASS)

Sidebar modules do not import each other. All cross-module communication goes through the event bus (`events.js`). `annotation-sidebar.js` is the sole orchestrator importing all 15 sidebar sub-modules.

### 2.2 annotation-sidebar.js still orchestrates too much (MEDIUM)

**Evidence:** 25 imports, 14 functions, 647 lines. While dramatically improved from 2,268 lines, it still owns: shadow DOM creation, tab switching, collapse/expand, badge counting, send logic, and refresh orchestration. The `create()` function alone is the largest single function.

### 2.3 WS_MESSAGES duplicate across workspaces (MEDIUM)

**Evidence:** `server/src/ws-message-types.js` and `extension/lib/ws-message-types.js` are identical except for the sync comment direction. No build-time or CI check enforces sync. Adding a message type to one side silently breaks the protocol.

## 4. Dead Code / Unused Exports

### 4.1 styles.js constants defined but underused (MEDIUM)

**Evidence:** `styles.js` exports 13 constants (`FONT`, `FONT_MONO`, `COLOR`, `LABEL_STYLE`, `TOGGLE_STYLE`, `TOGGLE_ON`, `TOGGLE_OFF`, `ACTION_BTN_STYLE`, `ICON_BTN_STYLE`, `TAB_STYLE`, `DESC_STYLE`, `DIVIDER_STYLE`, `DIVIDER_SUBTLE_STYLE`). All 13 sidebar files import from styles.js, but many still hardcode the same values inline. The constants exist but adoption is incomplete.

### 4.2 No other dead exports found (PASS)

The 2026-04-12 dead code elimination pass was thorough. No unused exports detected in the current scan.

## 5. Inconsistent Patterns

### 5.1 Tool register() signatures (MEDIUM)

**Evidence:** 5 different signatures across 37 tools:
- `(server, _indexer, capturesDir)` - 18 tools
- `(server, indexer, capturesDir)` - 11 tools
- `(server, indexer)` - 2 tools (list-captures, get-session-status)
- `(server, queue)` - 2 tools (request-capture, get-request-status)
- `(server, _indexer, capturesDir, options)` - 1 tool (resolve-annotation)

The `_indexer` vs `indexer` naming is inconsistent - some tools that don't use the indexer still receive it positionally.

### 5.2 F19 prompt injection wrapping gaps (HIGH - security overlap)

**Evidence:** 11 tools use `wrapCapturedText`/`wrapComment`. Per SRR-004, 8+ tools return raw page text without wrapping: `compare-captures`, `compare-baseline`, `get-latest`, `get-session`, `get-capture`, `check-consistency`, `get-page-summary`, `get-component-coverage`.

### 5.3 _notice field inconsistency (MEDIUM)

**Evidence:** 5 of 37 tools include a `_notice` field warning agents about untrusted text. Each uses different wording. The other 32 tools that return captured text do not include any notice.

### 5.4 innerHTML in sidebar (LOW - security overlap)

**Evidence:** 13 `innerHTML` assignments across sidebar files. All use hardcoded SVG strings (not exploitable), but violate the project's own policy that `icons.js` DOM API factories should be used instead.

## 6. Test Coverage Gaps

### 6.1 Server tools without dedicated tests (HIGH)

**16 of 37 tools have no test file:**
`analyze-journey`, `analyze-patterns`, `check-annotation-status`, `check-consistency`, `compare-baseline`, `compare-screenshots`, `detect-recurring-issues`, `diff-annotations`, `generate-spec`, `get-capture-stats`, `get-unresolved`, `list-baselines`, `list-sessions`, `set-baseline`, `validate-capture`, `visualize-flow`

These tools are exercised indirectly through integration tests, but have no dedicated unit tests for edge cases.

### 6.2 Extension modules without tests (LOW)

- `collector-utils.js` - shared helper, no dedicated test (tested indirectly via collector tests)
- `sidebar/diagnostics.js` - 302 lines, no test file
- `sidebar/toggles.js` - 166 lines, no test file

### 6.3 annotation-sidebar.js test monolith (MEDIUM)

The sidebar tests are split across 20+ files in `tests/unit/sidebar/`, which is good. However, `sidebar-misc.test.js` (7,966 bytes) and `sidebar-mcp.test.js` (8,149 bytes) are catch-all files that should be further decomposed.

## 7. Documentation Gaps

### 7.1 JSDoc coverage (MEDIUM)

Server tools and analysis modules have consistent JSDoc on `register()` and exported functions. Extension sidebar modules are mixed - some have JSDoc (`collector-utils.js`, `events.js`), others have none (`diagnostics.js`, `toggles.js`, `captures.js`).

### 7.2 No JSDoc on sidebar render functions

`renderCaptures()`, `renderDiagnostics()`, `renderReviewList()`, `createSettings()` - the largest and most complex functions - lack `@param` and `@returns` documentation.

---

## 8. Constants Scattered vs Centralized

### 8.1 CSS colors (HIGH - see 1.2)

76 hardcoded colors despite `COLOR` constant existing. Migration is incomplete.

### 8.2 Port 9876 hardcoded in scripts (LOW)

**Evidence:** `viewgraph-doctor.js` (5 occurrences), `viewgraph-init.js` (1), `viewgraph-status.js` (1). Scripts can't import from server constants, so each defines the port inline. Acceptable given scripts are standalone, but a `const DEFAULT_PORT = 9876` at the top of each would improve clarity.

### 8.3 ENV_ constants (PASS)

`server/src/constants.js` exports `ENV_CAPTURES_DIR`, `ENV_MAX_CAPTURES`, `ENV_HTTP_PORT`, `ENV_IDLE_TIMEOUT`, `ENV_ALLOWED_DIRS`. All env var access goes through these constants. Clean.

---

## 9. Error Handling Consistency

### 9.1 Tool error responses (GOOD)

Tools using `readAndParse` get consistent error handling with fuzzy filename suggestions. The 16 tools bypassing it have simpler but still consistent error patterns via `errorResponse()`.

### 9.2 Collector error boundary (PASS)

All collectors are wrapped by `safeCollect()` in `enrichment.js`. One collector failure never crashes the capture. Well-designed.

### 9.3 HTTP receiver error handling (PASS)

All `JSON.parse` calls in `http-receiver.js` are wrapped in try/catch. Path traversal prevention via `validateCapturePath`. Payload size limits enforced.

---

## 10. Naming Conventions

### 10.1 _indexer vs indexer (LOW)

18 tools use `_indexer` (unused parameter), 11 use `indexer`. Inconsistent but harmless - the underscore prefix correctly signals "unused."

### 10.2 File naming (PASS)

Consistent kebab-case across all source files. Test files mirror source structure. No naming drift detected.

### 10.3 Function naming (PASS)

Sidebar: `create*` for constructors, `render*` for DOM renderers. Server: `register()` for tool registration. Collectors: `collect*` for data gathering. Consistent.
