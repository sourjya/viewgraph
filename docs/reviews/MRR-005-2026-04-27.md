# MRR-005 - Maintainability Review Report (M19 Service Worker Migration)

**Date:** 2026-04-27
**Baseline:** MRR-004 (2026-04-27)
**Scope:** M19 service worker communication migration - 8 new/rewritten modules, 6 new test files, 2 new prompts
**Cross-reference:** SRR-006 (2026-04-27) - security review of same codebase

---

## Executive Summary

The M19 service worker migration is architecturally sound. The 5 new `sw/` modules are well-decomposed, each under 230 lines, with clear single responsibilities. Test coverage is strong: 83 new tests across 6 test files covering all 5 SW modules plus the transport-client proxy. The `transport-client.js` → `transport-handler.js` → `transport.js` delegation chain is clean and the operation whitelist is a good security boundary.

However, the migration introduced **three systemic duplication issues** that will cause maintenance drift:

1. **Triple-maintained OPS map** - The 14-operation transport routing table is defined independently in `transport-handler.js`, `transport-client.js`, and `tests/mocks/chrome.js`. Adding a new transport operation requires updating all three files.

2. **Duplicated storage key strings** - `WS_EVENTS_KEY`, `SERVER_URL_KEY`, `RESOLVED_KEY_PREFIX`, and `PENDING_KEY` are defined as string literals in multiple modules with no shared source of truth.

3. **Duplicated network constants** - `DEFAULT_HTTP_PORT`, `PORT_SCAN_RANGE`, and `SERVER_HOST` are redeclared in `discovery-sw.js` despite existing in `constants.js`.

The pre-existing **duplicate discoverServer block** in `annotation-sidebar.js` (MRR-004 finding 11.1) was NOT fixed by M19 and has grown slightly worse - the second block now has an M19 comment but still duplicates ~100 lines of discovery/status/trust logic.

**New findings:** 12 (1 HIGH, 6 MEDIUM, 5 LOW)
**Resolved from MRR-004:** 2 (11.2 dead lookupCapturesDir, 11.4 unused extVer - both addressed by M19 comments but code still present)

## New Findings

### 12.1 - Triple-maintained transport operation map

| Attribute | Value |
|---|---|
| **Severity** | HIGH |
| **Scope** | Systemic |
| **Phase** | Phase 2 |
| **Files** | `extension/lib/sw/transport-handler.js:20-33`, `extension/lib/transport-client.js:56-107`, `extension/tests/mocks/chrome.js:31-37` |

**Evidence:** The 14 transport operations (`getInfo`, `getHealth`, `getCaptures`, `sendCapture`, etc.) are independently defined in three places:

1. **transport-handler.js** - `OPS` object mapping op names to `transport.*` calls with arg extraction
2. **transport-client.js** - 14 individual exported functions that each call `_send(opName, args)`
3. **tests/mocks/chrome.js** - `ops` object inside `mockSendMessage` that mirrors the handler's routing

Adding a 15th operation (e.g., `getAnnotationContext`) requires editing all three files. If any one is missed, the operation silently fails with "Unknown transport op" in production or tests pass but the real handler rejects the call.

**Impact:** Change amplification (3 files per new operation). The mock is especially dangerous because it can drift from the handler without any test catching it - the mock IS the test infrastructure.

**Recommended refactor:** Extract a shared `TRANSPORT_OPS` array or map from a single source file (e.g., `lib/sw/transport-ops.js`). The handler, client, and mock all import from it. The handler uses it to build the dispatch table, the client uses it to generate exported functions, and the mock uses it to build the routing.

Minimal version: export just the operation names as an array. The handler keeps its arg-extraction logic, but the mock and client validate against the shared list.

**Effort:** 1-2 hours
**Regression risk:** Low - the API surface doesn't change, only the source of truth

### 12.2 - Duplicated chrome.storage key strings across SW and content script

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Repeated |
| **Phase** | Phase 1 |
| **Files** | `extension/lib/sw/ws-manager.js:17`, `extension/lib/transport-client.js:22`, `extension/lib/sw/sync-alarms.js:21-27`, `extension/lib/sidebar/sync.js:77-80`, `extension/lib/storage.js:24` |

**Evidence:** Storage key strings are defined as local constants in multiple modules:

| Key | Defined in |
|---|---|
| `'vg-ws-events'` | `ws-manager.js:17`, `transport-client.js:22`, `sync.js:117` (hardcoded string) |
| `'vg-server-url'` | `discovery-sw.js:34`, `transport-client.js:25` |
| `'vg-resolved-'` | `sync-alarms.js:21`, `sync.js:77` |
| `'vg-pending-requests'` | `sync-alarms.js:24`, `sync.js:80`, `storage.js:24` |

The SW writes to these keys and the content script reads from them. If a key string drifts between writer and reader, events silently stop flowing with no error.

**Recommended refactor:** Add all M19 storage keys to `extension/lib/storage.js` KEYS object (which already exists and defines `pendingRequests`). Import from there in all modules.

**Effort:** 20 minutes
**Regression risk:** None

### 12.3 - Redeclared network constants in discovery-sw.js

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Phase** | Phase 1 |
| **Files** | `extension/lib/sw/discovery-sw.js:22-28`, `extension/lib/constants.js:34-43` |

**Evidence:** `discovery-sw.js` declares its own copies of three constants that already exist in `constants.js`:

```js
// discovery-sw.js:22-28
const DEFAULT_HTTP_PORT = 9876;
const PORT_SCAN_RANGE = 4;
const SERVER_HOST = '127.0.0.1';
```

```js
// constants.js:34-40
export const DEFAULT_HTTP_PORT = 9876;
export const PORT_SCAN_RANGE = 4;
export const SERVER_HOST = '127.0.0.1';
```

The JSDoc even references `constants.js` as the source: `@see lib/constants.js - DEFAULT_HTTP_PORT, PORT_SCAN_RANGE, SERVER_HOST`.

**Impact:** If the port range changes (e.g., expanding to 5 ports), `constants.js` gets updated but `discovery-sw.js` keeps the old value.

**Recommended refactor:** Import from `constants.js`: `import { DEFAULT_HTTP_PORT, PORT_SCAN_RANGE, SERVER_HOST } from '#lib/constants.js';`

**Effort:** 5 minutes
**Regression risk:** None

### 12.5 - mockChrome missing `storage.session` in defaults

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Phase** | Phase 1 |
| **Files** | `extension/tests/mocks/chrome.js` |

**Evidence:** The shared `mockChrome()` helper provides `storage.local`, `storage.sync`, and `storage.onChanged` but does NOT include `storage.session`. The `auth-sw.test.js` file works around this by providing its own `storage.session` mock in the `overrides` parameter.

Any future test that imports a module touching `chrome.storage.session` (e.g., importing `auth-sw.js` transitively) will get `TypeError: Cannot read properties of undefined (reading 'get')` unless it manually adds the session mock.

**Recommended refactor:** Add `storage.session` to the default mock:
```js
session: {
  get: vi.fn((key) => Promise.resolve({})),
  set: vi.fn(() => Promise.resolve()),
  remove: vi.fn(() => Promise.resolve()),
  ...overrides.storage?.session,
},
```

**Effort:** 5 minutes
**Regression risk:** None - additive change

### 12.6 - mockChrome double-merge pattern is fragile

| Attribute | Value |
|---|---|
| **Severity** | MEDIUM |
| **Scope** | Local |
| **Phase** | Phase 2 |
| **Files** | `extension/tests/mocks/chrome.js:57-81` |

**Evidence:** The mock constructs the object with spread overrides inside the literal, then re-merges overrides AFTER construction:

```js
const mock = {
  storage: {
    local: { ...defaults, ...overrides.storage?.local },
    ...overrides.storage,  // ← this clobbers local if overrides.storage has a local key
  },
};
// Then AGAIN:
if (overrides.storage?.local) mock.storage.local = { ...mock.storage.local, ...overrides.storage.local };
```

The `...overrides.storage` spread at line 70 can clobber the carefully merged `local` object if `overrides.storage` contains a `local` key. The post-construction merge at line 79 then re-applies it. This was flagged in the v0.6.0 changelog as a bug fix ("...overrides at top level was clobbering nested objects") but the fix is a band-aid - the double-merge is confusing and the `...overrides.storage` spread still runs.

Additionally, `storage.sync` and `storage.session` are NOT re-merged post-construction, so overrides to those are applied inconsistently compared to `storage.local`.

**Recommended refactor:** Remove the inline `...overrides.storage` spread entirely. Build the mock with defaults only, then deep-merge overrides in a single pass at the end. Or use a simple `deepMerge(defaults, overrides)` utility.

**Effort:** 30 minutes
**Regression risk:** Medium - must verify all 12+ test files using mockChrome

### 12.9 - discovery.js getAllServers() is a degraded stub

| Attribute | Value |
|---|---|
| **Severity** | LOW |
| **Scope** | Local |
| **Phase** | Phase 2 |
| **Files** | `extension/lib/discovery.js:33-40` |

**Evidence:** The rewritten `getAllServers()` sends a `vg-get-server` message (which returns a single matched server) and wraps it in an array. The comment acknowledges this: "vg-get-server returns a single match; for getAllServers we need the full list. Fall back to the fetch-info proxy for now."

| **Files** | `discovery-sw.js` (4), `auth-sw.js` (4), `sync-alarms.js` (4), `ws-manager.js` (3) |

**Evidence:** 15 `catch { }` blocks across the 5 SW modules. Most have inline comments (`/* tests */`, `/* server offline */`, `/* no cached registry */`) which is acceptable. However, 3 have no comment at all:

## Positive Observations

1. **Clean module decomposition.** Each SW module has a single responsibility: transport routing, discovery, auth, WebSocket lifecycle, alarm-based sync. No module exceeds 230 lines. This is textbook service worker architecture.

2. **Strong test coverage for new code.** 83 tests across 6 files covering all 5 SW modules + transport-client. Test-to-source ratio is 1.13:1 (1139 test lines / 1009 source lines). Edge cases like offline servers, WS constructor failures, and chrome.runtime.lastError are covered.

3. **Security boundary is well-defined.** The `OPS` whitelist in `transport-handler.js` prevents content scripts from invoking arbitrary transport methods. The content script has zero direct HTTP/WS connections after M19.

4. **Backward compatibility preserved.** `discovery.js` maintains the same API surface (`discoverServer`, `getAllServers`, `getAgentName`, `resetServerCache`). Existing consumers don't need changes.

5. **WebSocket lifecycle is reference-counted.** `ws-manager.js` correctly handles multiple sidebars across tabs with a single connection. The keepalive/reconnect logic is clean.

6. **url-utils.js is properly shared.** `discovery-sw.js` imports URL normalization from the shared `url-utils.js` module instead of reimplementing it.

## Consolidation Themes

### Theme A: Storage Key Centralization
Findings 12.2, 12.8. The M19 migration introduced a new communication channel (chrome.storage) between SW and content script. The storage keys are the contract between writer and reader. They must be centralized.

### Theme B: Transport Operation Registry
Finding 12.1. The transport operation list is the API contract between content script and service worker. It's currently maintained in 3 places. This is the highest-leverage M19-specific fix.

### Theme C: Discovery Consolidation (carried)
Findings 12.3, 12.4, 12.9. Discovery logic is now split across `discovery-sw.js` (SW), `discovery.js` (content script client), and `annotation-sidebar.js` (duplicate inline block). The SW owns discovery but the content script still has redundant logic.
